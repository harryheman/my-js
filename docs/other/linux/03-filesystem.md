# Этап 2. Файловая система

## Всё — файл

В Linux есть фундаментальный принцип: **всё является файлом**. Обычные файлы, каталоги,
устройства (диски, клавиатура), сетевые сокеты — система представляет их как файлы.
Это звучит странно, но на практике означает одно: для работы со всем этим используются
одни и те же инструменты.

Ещё одно важное отличие от Windows: в Linux нет дисков `C:`, `D:` и так далее.
Вся файловая система начинается с единственного корня — `/`. Все диски, флешки
и сетевые хранилища «подключаются» (монтируются) как каталоги внутри этого дерева.

## Структура каталогов

В главе 01 мы выполнили `ls /` и увидели набор каталогов в корне. Это не случайный
набор — в Linux существует стандарт **FHS** (Filesystem Hierarchy Standard), который
определяет, что где лежит.

Вот основные каталоги, с которыми вы будете сталкиваться:

| Каталог | Назначение |
|---------|-----------|
| `/` | корень файловой системы, начало всего дерева |
| `/home` | домашние каталоги пользователей (`/home/student`, `/home/admin` и т.д.) |
| `/root` | домашний каталог суперпользователя root |
| `/etc` | конфигурационные файлы системы и программ |
| `/var` | изменяемые данные: логи, почта, базы данных |
| `/tmp` | временные файлы (очищается при перезагрузке) |
| `/usr` | программы и библиотеки, установленные в системе |
| `/bin` | основные исполняемые файлы (команды) |
| `/sbin` | системные команды (обычно для администратора) |
| `/opt` | дополнительное программное обеспечение |
| `/dev` | файлы устройств (диски, терминалы и др.) |
| `/proc` | виртуальная файловая система с информацией о процессах и ядре |

> **Привязка к проекту.** Когда мы будем устанавливать nginx и WordPress, вы увидите
> эти каталоги в деле:
> - конфигурация nginx — в `/etc/nginx/`
> - логи nginx — в `/var/log/nginx/`
> - файлы сайта — в `/var/www/`
>
> Пока просто запомните, что конфиги живут в `/etc`, логи — в `/var/log`,
> а данные сайтов — в `/var/www`.

## Абсолютные и относительные пути

В Linux есть два способа указать расположение файла или каталога.

**Абсолютный путь** начинается с `/` и описывает полный маршрут от корня:

```
/home/student/documents/notes.txt
```

**Относительный путь** отсчитывается от текущего каталога и не начинается с `/`:

```
documents/notes.txt
```

Если вы находитесь в `/home/student`, оба пути указывают на один и тот же файл.

Для относительных путей используются специальные обозначения:

| Обозначение | Значение |
|-------------|----------|
| `.` | текущий каталог |
| `..` | родительский каталог (на уровень вверх) |
| `~` | домашний каталог текущего пользователя |

Примеры:

```bash
# Подняться на уровень вверх
cd ..

# Подняться на два уровня вверх
cd ../..

# Перейти в соседний каталог (вверх, потом вниз)
cd ../other-directory
```

## Навигация

### cd — перемещение по каталогам (подробнее)

В главе 01 мы уже использовали `cd`. Вот полезные приёмы:

```bash
# Перейти в каталог по абсолютному пути
cd /var/log

# Подняться на уровень вверх
cd ..

# Вернуться в домашний каталог
cd
# или
cd ~

# Вернуться в предыдущий каталог (где были до последнего cd)
cd -
```

Последний вариант (`cd -`) особенно удобен: он работает как кнопка «назад»
и переключает вас между двумя каталогами.

### ls — просмотр содержимого каталога (подробнее)

В главе 01 мы использовали `ls` без параметров. Теперь разберём полезные флаги.

**`ls -l`** — подробный (длинный) формат:

```bash
ls -l /etc
```

```
drwxr-xr-x 2 root root 4096 янв 15 10:30 apt
-rw-r--r-- 1 root root 2981 фев  3 09:15 bash.bashrc
-rw-r--r-- 1 root root  367 мар 10 14:22 hostname
```

Разберём вывод по столбцам:

| Столбец | Пример | Значение |
|---------|--------|----------|
| 1 | `drwxr-xr-x` | тип файла и права доступа (подробно — в главе 03) |
| 2 | `2` | количество жёстких ссылок |
| 3 | `root` | владелец |
| 4 | `root` | группа |
| 5 | `4096` | размер в байтах |
| 6 | `янв 15 10:30` | дата последнего изменения |
| 7 | `apt` | имя файла или каталога |

Первый символ в столбце прав показывает тип:
- `d` — каталог (directory)
- `-` — обычный файл
- `l` — символическая ссылка (аналог ярлыка в Windows)

**`ls -a`** — показать скрытые файлы:

```bash
ls -a ~
```

```
.  ..  .bash_history  .bashrc  .profile
```

В Linux файлы, имя которых начинается с точки (`.`), считаются **скрытыми**.
Обычный `ls` их не показывает. Это не защита — просто способ не засорять вывод
служебными файлами.

**`ls -la`** — подробный формат + скрытые файлы (комбинация флагов):

```bash
ls -la ~
```

**`ls -lh`** — подробный формат с читаемыми размерами (human-readable):

```bash
ls -lh /var/log
```

```
-rw-r----- 1 root adm  15K мар 10 14:00 auth.log
-rw-r--r-- 1 root root 248K мар  9 06:25 dpkg.log
```

Вместо `15360` показывается `15K`, вместо `253952` — `248K`. Размеры выводятся
в килобайтах (K), мегабайтах (M) или гигабайтах (G).

> **Совет.** Флаги можно комбинировать: `ls -la`, `ls -lah`. Порядок флагов
> не важен: `ls -la` и `ls -al` — одно и то же.

## Создание файлов и каталогов

### touch — создать пустой файл

```bash
touch myfile.txt
```

Если файл уже существует, `touch` обновит его дату последнего изменения, но
не изменит содержимое.

### mkdir — создать каталог

```bash
mkdir projects
```

Если нужно создать вложенную структуру каталогов, где промежуточные каталоги
ещё не существуют, используйте флаг `-p`:

```bash
mkdir -p projects/website/config
```

Без `-p` команда выдаст ошибку, если каталог `projects/website` не существует.
С `-p` она создаст всю цепочку.

### Практика: структура каталогов для проекта

Подготовим структуру каталогов для нашего будущего сайта на WordPress. Пока мы
создадим её в домашнем каталоге — для тренировки:

```bash
cd ~
mkdir -p wordpress-project/config
mkdir -p wordpress-project/logs
mkdir -p wordpress-project/site
```

Проверим результат:

```bash
ls -la wordpress-project/
```

```
drwxr-xr-x 2 student student 4096 мар 10 15:00 config
drwxr-xr-x 2 student student 4096 мар 10 15:00 logs
drwxr-xr-x 2 student student 4096 мар 10 15:00 site
```

Создадим несколько файлов для экспериментов:

```bash
touch wordpress-project/config/nginx.conf
touch wordpress-project/config/php.ini
touch wordpress-project/logs/access.log
touch wordpress-project/logs/error.log
touch wordpress-project/site/index.html
```

## Копирование, перемещение, удаление

### cp — копирование

Копирование файла:

```bash
cp wordpress-project/config/nginx.conf wordpress-project/config/nginx.conf.backup
```

Копирование каталога целиком — обязательно с флагом `-r` (recursive):

```bash
cp -r wordpress-project/config wordpress-project/config-backup
```

Без `-r` команда `cp` откажется копировать каталог и выдаст ошибку.

### mv — перемещение и переименование

Переименование файла (перемещение в тот же каталог, но с другим именем):

```bash
mv wordpress-project/site/index.html wordpress-project/site/index.php
```

Перемещение файла в другой каталог:

```bash
mv wordpress-project/site/index.php wordpress-project/
```

Перемещение с одновременным переименованием:

```bash
mv wordpress-project/index.php wordpress-project/site/main.php
```

> **Важно.** `mv` в Linux не спрашивает подтверждения. Если файл с таким именем
> уже существует в целевом каталоге — он будет перезаписан без предупреждения.

### rm — удаление

Удаление файла:

```bash
rm wordpress-project/site/main.php
```

Удаление каталога со всем содержимым — флаг `-r` (recursive):

```bash
rm -r wordpress-project/config-backup
```

Удаление пустого каталога (альтернатива):

```bash
rmdir wordpress-project/logs
```

`rmdir` удалит каталог только если он пуст. Если внутри есть файлы — будет ошибка.

> **Осторожно с `rm -rf`!**
>
> Команда `rm -rf` удаляет каталог со всем содержимым **без запроса подтверждения**
> и **без возможности восстановления**. В Linux нет «Корзины» в командной строке.
> Удалённые файлы восстановить крайне сложно.
>
> Особенно опасно:
> ```bash
> # НИКОГДА НЕ ДЕЛАЙТЕ ЭТО:
> rm -rf /
> ```
> Эта команда уничтожит всю файловую систему. Современные системы защищены от этого
> конкретного случая, но вариации вроде `rm -rf /*` по-прежнему опасны.
>
> Правило: **всегда дважды проверяйте путь перед выполнением `rm -rf`**.

## Просмотр содержимого файлов

### cat — вывести файл целиком

```bash
cat /etc/hostname
```

```
debian
```

`cat` выводит всё содержимое файла разом. Для коротких файлов это удобно, но для
больших файлов (например, логов на тысячи строк) экран заполнится текстом, и вы
увидите только конец файла.

### less — постраничный просмотр

```bash
less /etc/bash.bashrc
```

`less` открывает файл в режиме чтения, как `man`. Навигация такая же:

| Клавиша | Действие |
|---------|----------|
| `Пробел` / `Page Down` | следующая страница |
| `b` / `Page Up` | предыдущая страница |
| стрелки `вверх` / `вниз` | прокрутка по строкам |
| `/текст` + Enter | поиск |
| `n` | следующее совпадение |
| `q` | выход |

Используйте `less` для просмотра больших файлов — логов, конфигураций, исходного кода.

### head — начало файла

```bash
head /etc/passwd
```

Выводит первые 10 строк файла. Можно указать другое количество:

```bash
head -n 5 /etc/passwd
```

Выведет первые 5 строк.

### tail — конец файла

```bash
tail /etc/passwd
```

Выводит последние 10 строк. Аналогично можно указать количество:

```bash
tail -n 3 /etc/passwd
```

`tail` особенно полезен для просмотра логов — свежие записи добавляются в конец файла:

```bash
tail /var/log/syslog
```

> **Полезный приём.** Команда `tail -f` позволяет следить за файлом в реальном времени —
> новые строки будут появляться на экране по мере их добавления. Это незаменимо
> при отладке работающих сервисов:
> ```bash
> tail -f /var/log/syslog
> ```
> Для выхода нажмите `Ctrl+C`.

## Врезка: кодировки и переводы строк

Если вы когда-нибудь переносили текстовые файлы с Windows на Linux (или наоборот),
то могли столкнуться со странными символами или «слипшимися» строками. Вот почему
это происходит.

### Кодировки

**Кодировка** — это таблица соответствия между символами и числами. Компьютер хранит
только числа, и кодировка говорит ему, какое число соответствует какой букве.

В Windows исторически использовались разные кодировки для разных языков: CP1251 для
русского, CP1252 для западноевропейских и так далее. В Linux стандартом является
**UTF-8** — универсальная кодировка, которая поддерживает символы всех языков мира.

Если открыть файл в неправильной кодировке, вместо русского текста будут «кракозябры».
Решение — убедиться, что и отправитель, и получатель используют одну кодировку.
Сегодня лучший выбор — всегда использовать UTF-8.

### Переводы строк

В конце каждой строки текстового файла стоит невидимый символ перевода строки.
Проблема в том, что Windows и Linux используют **разные символы**:

| Система | Символ перевода строки | Обозначение |
|---------|----------------------|-------------|
| Linux / macOS | LF (Line Feed) | `\n` |
| Windows | CR+LF (Carriage Return + Line Feed) | `\r\n` |

Когда вы переносите файл из Windows в Linux, лишний символ `\r` (CR) может вызвать
проблемы. Например, bash-скрипт, написанный в Windows, может не запускаться на Linux
с ошибкой `\r: command not found`.

### Как проверить

Команда `file` определяет тип файла и показывает информацию о кодировке и формате
переводов строк:

```bash
file /etc/hostname
```

```
/etc/hostname: ASCII text
```

Если файл создан в Windows, вы увидите:

```
script.sh: Bash script, ASCII text, with CRLF line terminators
```

Фраза **CRLF line terminators** означает, что файл использует Windows-формат
переводов строк.

> **Совет.** Если вы редактируете файлы в Windows и переносите их на Linux-сервер,
> используйте редактор, который умеет сохранять в формате LF (например, VS Code —
> в строке состояния можно переключить CRLF на LF).

## Практическое задание

1. Перейдите в домашний каталог. Создайте структуру каталогов:
   ```
   ~/project/
   ├── backup/
   ├── config/
   ├── data/
   │   ├── images/
   │   └── uploads/
   └── logs/
   ```
   Подсказка: используйте `mkdir -p`.

2. Создайте файлы:
   - `~/project/config/app.conf`
   - `~/project/data/readme.txt`
   - `~/project/logs/app.log`

3. Скопируйте файл `app.conf` в каталог `backup/` с именем `app.conf.bak`.

4. Переименуйте `readme.txt` в `README.txt`.

5. Скопируйте каталог `config/` целиком в `backup/` (чтобы получился `backup/config/`).

6. Удалите файл `app.log`.

7. Посмотрите содержимое файла `/etc/os-release` командой `cat`. Какой дистрибутив
   и какая версия установлены?

8. Посмотрите первые 5 строк файла `/etc/passwd` с помощью `head`.

9. Выполните `file /etc/passwd` — какой тип файла и кодировку показывает команда?

10. Проверьте результат: выполните `ls -laR ~/project/` (флаг `-R` выводит содержимое
    всех подкаталогов рекурсивно).
