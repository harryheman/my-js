# Этап 10. Сеть: ip, ss, firewall (ufw), основы DNS

## Зачем разбираться в сети

Веб-сервер без сети — как телефон без связи. В главе 11 мы установим
nginx и откроем сайт миру, но сначала нужно понять, как Linux работает
с сетью: какой у машины IP-адрес, какие порты открыты, кто к ним
подключается и как защитить сервер файрволом.

В Windows вы, возможно, пользовались командой `ipconfig` или заходили
в «Центр управления сетями». В Linux те же задачи решаются
из командной строки — и здесь это даже удобнее.

> **Привязка к проекту.** В этой главе мы узнаем IP-адрес нашего
> сервера, научимся проверять сетевые соединения и настроим файрвол,
> открыв порты для SSH (22) и будущего веб-сервера (80, 443).
> Всё это понадобится в главе 11, когда будем запускать nginx.

## Врезка: основы IP-сетей

Прежде чем переходить к командам, разберёмся с базовыми понятиями.
Если вы уже знакомы с IP-адресами и портами — можете пропустить
этот раздел.

### IP-адреса

Каждое устройство в сети имеет уникальный **IP-адрес** — как почтовый
адрес для компьютера. В формате IPv4 он выглядит так:

```
192.168.1.100
```

Четыре числа от 0 до 255, разделённые точками.

IP-адреса делятся на **частные** (используются внутри локальной сети)
и **публичные** (видны в интернете):

| Диапазон | Тип | Пример использования |
|----------|-----|---------------------|
| `10.0.0.0` – `10.255.255.255` | Частный | Корпоративные сети |
| `172.16.0.0` – `172.31.255.255` | Частный | Облачные провайдеры |
| `192.168.0.0` – `192.168.255.255` | Частный | Домашние сети |
| Всё остальное | Публичный | Серверы в интернете |

Ваша виртуальная машина, скорее всего, имеет частный адрес
(например, `192.168.x.x` или `10.x.x.x`).

Особый адрес **127.0.0.1** (localhost) — это «сам компьютер». Он
всегда указывает на текущую машину.

### Маска подсети и CIDR

**Маска подсети** определяет, какая часть адреса обозначает сеть,
а какая — конкретный компьютер в ней. Записывается двумя способами:

```
255.255.255.0      — классическая запись
/24                — CIDR-нотация (то же самое)
```

Запись `192.168.1.0/24` означает: сеть `192.168.1.x`, где последнее
число (от 0 до 255) — адрес компьютера. В такой сети может быть
до 254 устройств.

Для нашего курса достаточно знать, что `/24` — самая
распространённая маска для небольших сетей.

### Шлюз (gateway)

**Шлюз** — это «дверь наружу» из локальной сети. Когда компьютер
хочет связаться с адресом за пределами своей подсети, он отправляет
пакет через шлюз. Обычно это маршрутизатор (роутер) с адресом вроде
`192.168.1.1`.

### Порты

IP-адрес определяет компьютер, а **порт** — конкретную программу
на нём. Порт — это число от 1 до 65535.

Когда вы подключаетесь по SSH, ваш клиент соединяется с адресом
сервера на **порт 22**. Когда откроете сайт в браузере — браузер
обратится на **порт 80** (HTTP) или **443** (HTTPS).

Известные порты:

| Порт | Сервис |
|------|--------|
| 22 | SSH |
| 80 | HTTP (веб-сервер) |
| 443 | HTTPS (веб-сервер, шифрованное соединение) |
| 3306 | MySQL / MariaDB |
| 5432 | PostgreSQL |

Аналогия: IP-адрес — это адрес здания, а порт — номер квартиры.
Почтальон (сетевой пакет) приходит по адресу и стучится в нужную
квартиру.

### TCP и UDP

Два основных протокола передачи данных:

- **TCP** — надёжный: гарантирует доставку и порядок пакетов.
  Используется для SSH, HTTP, баз данных — всего, где важна
  целостность данных.
- **UDP** — быстрый, но без гарантий: пакеты могут потеряться.
  Используется для DNS-запросов, видео, игр — где скорость важнее
  надёжности.

В большинстве серверных задач вы будете работать с TCP.

## ip: сетевые интерфейсы и адреса

Команда `ip` — главный инструмент для просмотра и настройки сети
в Linux. Она заменила устаревшую `ifconfig`.

### Посмотреть IP-адреса

```bash
ip addr show
```

Или короче:

```bash
ip a
```

Пример вывода:

```
1: lo: <LOOPBACK,UP,LOWER_UP> mtu 65536
    inet 127.0.0.1/8 scope host lo
2: eth0: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1500
    inet 192.168.1.100/24 brd 192.168.1.255 scope global eth0
```

Что здесь важно:

| Элемент | Значение |
|---------|----------|
| `lo` | Loopback — виртуальный интерфейс «на себя» (127.0.0.1) |
| `eth0` | Сетевой интерфейс (может называться `ens3`, `enp0s3` и т.д.) |
| `UP` | Интерфейс включён и работает |
| `inet 192.168.1.100/24` | IP-адрес и маска подсети |

> **Названия интерфейсов.** В разных системах сетевой интерфейс
> может называться по-разному: `eth0`, `ens3`, `enp0s3`. Это зависит
> от системы именования. В примерах мы используем `eth0`, но
> у вас имя может отличаться — это нормально.

### Посмотреть маршруты

```bash
ip route show
```

Или короче:

```bash
ip r
```

```
default via 192.168.1.1 dev eth0
192.168.1.0/24 dev eth0 proto kernel scope link src 192.168.1.100
```

Первая строка — **шлюз по умолчанию** (`192.168.1.1`). Все пакеты
«наружу» отправляются через него.

### Посмотреть интерфейсы

```bash
ip link show
```

Показывает список сетевых интерфейсов и их состояние (UP/DOWN),
без IP-адресов.

## ss: сетевые соединения

Команда `ss` (socket statistics) показывает, какие программы слушают
сетевые порты и какие соединения установлены. Она заменила устаревшую
`netstat`.

### Какие порты слушают

Самая полезная комбинация флагов:

```bash
ss -tlnp
```

| Флаг | Значение |
|------|----------|
| `-t` | Только TCP-соединения |
| `-l` | Только слушающие (listening) порты |
| `-n` | Показывать номера портов, а не имена сервисов |
| `-p` | Показывать имя процесса (нужен sudo для чужих процессов) |

Пример вывода:

```
State   Recv-Q  Send-Q  Local Address:Port  Peer Address:Port  Process
LISTEN  0       128     0.0.0.0:22           0.0.0.0:*         users:(("sshd",pid=654,fd=3))
```

Здесь видно: процесс `sshd` слушает порт 22 на всех интерфейсах
(`0.0.0.0`). Когда мы установим nginx, здесь появится ещё и порт 80.

Чтобы увидеть процессы всех пользователей:

```bash
sudo ss -tlnp
```

### Установленные соединения

```bash
ss -tn
```

Покажет активные TCP-соединения (без `-l` — только установленные,
не слушающие). Вы увидите своё SSH-подключение.

## Проверка соединений

### ping — доступность хоста

```bash
ping 8.8.8.8
```

Отправляет пакеты и показывает время ответа. `8.8.8.8` — публичный
DNS-сервер Google, удобен для проверки доступа в интернет.

```
PING 8.8.8.8 (8.8.8.8) 56(84) bytes of data.
64 bytes from 8.8.8.8: icmp_seq=1 ttl=118 time=5.42 ms
64 bytes from 8.8.8.8: icmp_seq=2 ttl=118 time=5.38 ms
```

Остановить — `Ctrl+C`. Или указать количество пакетов:

```bash
ping -c 3 8.8.8.8      # отправить 3 пакета и остановиться
```

Если ответа нет — хост недоступен, или между вами файрвол блокирует
ICMP-пакеты.

### curl — HTTP-запрос

`curl` отправляет HTTP-запрос к серверу. Полезно для проверки
веб-сервисов:

```bash
curl http://example.com
```

Показывает HTML-код страницы. Чтобы увидеть только заголовки ответа:

```bash
curl -I http://example.com
```

```
HTTP/1.1 200 OK
Content-Type: text/html; charset=UTF-8
```

> **Привязка к проекту.** В главе 11, когда установим nginx,
> мы будем проверять его работу именно через `curl http://localhost`.

## DNS: разрешение имён

### Что такое DNS

Компьютеры общаются по IP-адресам, но людям удобнее запоминать имена:
`google.com` вместо `142.250.74.46`. Превращением имён в адреса
занимается **DNS** (Domain Name System) — своего рода телефонная
книга интернета.

Когда вы набираете `ping google.com`, система сначала спрашивает
DNS-сервер: «Какой IP у google.com?» — получает ответ и только потом
отправляет пакет.

### /etc/resolv.conf — настройки DNS

Файл `/etc/resolv.conf` содержит адрес DNS-сервера, к которому
обращается система:

```bash
cat /etc/resolv.conf
```

```
nameserver 192.168.1.1
```

Здесь `nameserver` — адрес DNS-сервера (часто совпадает со шлюзом).

### /etc/hosts — локальные записи

Файл `/etc/hosts` позволяет задать соответствие «имя → IP» вручную,
минуя DNS. Система проверяет его **раньше**, чем обращается
к DNS-серверу:

```bash
cat /etc/hosts
```

```
127.0.0.1       localhost
127.0.1.1       debian
```

Это удобно для разработки: можно добавить запись вроде
`127.0.0.1 mysite.local` и обращаться к локальному серверу по имени.

### Проверка DNS

Команда `host` показывает, в какой IP-адрес разрешается доменное имя:

```bash
host google.com
```

```
google.com has address 142.250.74.46
google.com has IPv6 address 2a00:1450:4010:c05::71
google.com mail is handled by 10 smtp.google.com.
```

Более подробная информация — через `dig` (если установлен):

```bash
dig google.com
```

## ufw: файрвол

### Зачем нужен файрвол

Файрвол (межсетевой экран) контролирует, какой сетевой трафик
разрешён, а какой заблокирован. По умолчанию сервер принимает
подключения на любой открытый порт. Файрвол позволяет ограничить
доступ: например, разрешить SSH (порт 22) и HTTP (порт 80), но
заблокировать всё остальное.

> **ufw и nftables.** В ядре Linux за фильтрацию трафика отвечает
> подсистема **nftables** — мощный, но сложный инструмент с собственным
> языком правил. **ufw** (Uncomplicated Firewall) — удобная обёртка
> над nftables, аналогично тому, как `adduser` — обёртка над `useradd`.
> Для повседневных задач ufw более чем достаточен.

### Установка и включение

В Debian ufw может быть не установлен по умолчанию:

```bash
sudo apt install ufw
```

Перед включением файрвола **обязательно** разрешите SSH, иначе
вы потеряете доступ к серверу:

```bash
sudo ufw allow 22
sudo ufw enable
```

> **Внимание!** Если вы включите ufw, не разрешив порт 22,
> ваше SSH-соединение будет разорвано, и вы не сможете подключиться
> к серверу. Всегда разрешайте SSH перед включением файрвола.

### Статус и правила

```bash
sudo ufw status
```

```
Status: active

To                         Action      From
--                         ------      ----
22                         ALLOW       Anywhere
```

Подробный вывод с номерами правил:

```bash
sudo ufw status numbered
```

### Разрешение и запрет

```bash
sudo ufw allow 80          # разрешить HTTP
sudo ufw allow 443         # разрешить HTTPS
sudo ufw allow 80/tcp      # разрешить только TCP на порт 80
```

Можно использовать имена сервисов:

```bash
sudo ufw allow ssh         # то же, что allow 22
sudo ufw allow http        # то же, что allow 80
```

Запретить порт:

```bash
sudo ufw deny 3306         # запретить доступ к MySQL снаружи
```

### Удаление правил

```bash
sudo ufw status numbered   # посмотреть номера правил
sudo ufw delete 3          # удалить правило номер 3
```

Или по описанию:

```bash
sudo ufw delete allow 80
```

### Отключение файрвола

```bash
sudo ufw disable
```

Правила сохраняются — при следующем `ufw enable` они вернутся.

## Практика: настраиваем сеть для проекта

### Шаг 1. Узнаём IP-адрес сервера

```bash
ip addr show
```

Найдите адрес вашего сетевого интерфейса (не `127.0.0.1`). Запишите
его — он понадобится в главе 11 для доступа к nginx.

### Шаг 2. Проверяем шлюз и доступ в интернет

```bash
ip route show
ping -c 3 8.8.8.8
```

Если `ping` работает — у вас есть доступ в интернет (нужен для
установки пакетов).

### Шаг 3. Смотрим, что слушает сеть

```bash
sudo ss -tlnp
```

Вы должны увидеть как минимум `sshd` на порту 22. Запомните эту
команду — в главе 11 вы будете проверять, что nginx слушает порт 80.

### Шаг 4. Настраиваем файрвол

```bash
sudo apt install ufw
sudo ufw allow 22          # SSH — обязательно первым!
sudo ufw allow 80          # HTTP — для будущего nginx
sudo ufw allow 443         # HTTPS — для будущего nginx
sudo ufw enable
sudo ufw status
```

Проверяем, что SSH по-прежнему работает (откройте второй терминал
и попробуйте подключиться).

### Шаг 5. Проверяем DNS

```bash
host google.com
cat /etc/resolv.conf
```

Убедитесь, что DNS работает — имена разрешаются в IP-адреса.

## Справочник команд

### Сетевые интерфейсы и адреса

|---------|----------|
| `ip addr show` (или `ip a`) | Показать IP-адреса интерфейсов |
| `ip route show` (или `ip r`) | Показать таблицу маршрутов и шлюз |
| `ip link show` | Показать сетевые интерфейсы |

### Соединения и диагностика

| Команда | Описание |
|---------|----------|
| `ss -tlnp` | Показать слушающие TCP-порты с процессами |
| `ss -tn` | Показать установленные TCP-соединения |
| `ping -c 3 адрес` | Проверить доступность хоста (3 пакета) |
| `curl URL` | Отправить HTTP-запрос |
| `curl -I URL` | Показать только HTTP-заголовки |

### DNS

| Команда | Описание |
|---------|----------|
| `host имя` | Узнать IP-адрес по доменному имени |

### Файрвол (ufw)

| Команда | Описание |
|---------|----------|

| `sudo ufw enable` | Включить файрвол |
| `sudo ufw disable` | Отключить файрвол |

| `sudo ufw deny порт` | Запретить порт |
| `sudo ufw delete номер` | Удалить правило по номеру |

## Практические задания

1. Выполните `ip addr show`. Найдите IP-адрес вашего сервера
   (не 127.0.0.1). В какой подсети он находится (посмотрите маску)?

2. Выполните `ip route show`. Какой шлюз по умолчанию (default via)?
   Выполните `ping -c 3` на этот шлюз — отвечает ли он?

3. Выполните `sudo ss -tlnp`. Какие сервисы слушают порты? На каких
   портах? Соответствует ли это тому, что вы ожидаете?

4. Выполните `ping -c 3 8.8.8.8`. Какое среднее время ответа?
   Теперь выполните `ping -c 3 google.com`. Если работает —
   значит, DNS настроен правильно.

5. Посмотрите файл `/etc/resolv.conf`. Какой DNS-сервер используется?
   Проверьте его работу командой `host ya.ru`.

6. Установите ufw (`sudo apt install ufw`). Разрешите порт 22,
   затем включите файрвол. Проверьте, что SSH по-прежнему работает
   (откройте новое подключение, не закрывая текущее!).

7. Добавьте правила для портов 80 и 443. Посмотрите список правил
   с номерами (`sudo ufw status numbered`). Удалите правило для
   порта 443, затем добавьте его заново.

8. Добавьте в файл `/etc/hosts` строку `127.0.0.1 mysite.local`
   (потребуется sudo). Проверьте командой `ping -c 1 mysite.local`,
   что имя разрешается в 127.0.0.1.
