# Этап 14. Docker: контейнеризация проекта

## Зачем нужны контейнеры

В главе 11 мы установили nginx, PHP-FPM и MariaDB вручную:
добавляли пакеты, редактировали конфиги, создавали пользователей
базы данных. Всё заработало — но представьте, что нужно повторить
это на втором сервере. Или на десятом. Или что после обновления
PHP сломался WordPress, а откатить изменения непросто.

Знакомая ситуация в разработке: «У меня на машине всё работает,
а на сервере — нет». Причина обычно одна — **различия в окружении**:
другая версия PHP, другие пути к файлам, не хватает библиотеки.

**Контейнер** решает эту проблему. Это изолированная среда, в которой
упакованы приложение и все его зависимости. Контейнер работает
одинаково на любой машине — на вашем ноутбуке, на тестовом сервере,
в продакшене.

**Docker** — самый популярный инструмент для работы с контейнерами.

### Контейнер vs. виртуальная машина

В главе 01 мы упоминали виртуальные машины — это полноценный
компьютер внутри компьютера со своей операционной системой.
Контейнер устроен проще:

```
  Виртуальная машина              Контейнер

┌───────────────────┐        ┌───────────────────┐
│    Приложение     │        │    Приложение     │
│    Библиотеки     │        │    Библиотеки     │
│    Гостевая ОС    │        └─────────┬─────────┘
└─────────┬─────────┘                  │
     Гипервизор                  Docker Engine
          │                            │
     Ядро хоста                   Ядро хоста
```

Контейнер не содержит отдельной ОС — он использует ядро хост-системы.
Поэтому контейнеры:

- **запускаются за секунды** (а не за минуты, как виртуальная машина);
- **потребляют меньше ресурсов** — нет накладных расходов на
  гостевую ОС;
- **легко переносимы** — образ контейнера можно запустить на любом
  сервере с Docker.

> **Один контейнер — один процесс.** Иногда новички пытаются
> использовать контейнер как виртуальную машину: запускают в одном
> контейнере и nginx, и PHP-FPM, и базу данных, ставят супервизор
> процессов. Технически это возможно, но это примерно как забивать
> шурупы молотком — работает, но неправильно. Вся суть контейнеров
> в том, что каждый из них отвечает за **один сервис**. Это упрощает
> обновление, масштабирование и отладку. Как запустить несколько
> контейнеров вместе — разберём в следующей главе.

> **Привязка к проекту.** В этой главе мы научимся запускать nginx
> из нашего wordpress-project в контейнере Docker. А в следующей
> главе соберём весь стек (nginx + PHP-FPM + WordPress + MariaDB)
> из нескольких связанных контейнеров.

## Врезка: форматы конфигурационных файлов

В предыдущих главах мы уже работали с конфигурационными файлами:
`nginx.conf`, `php.ini`, `/etc/apt/sources.list`. Прежде чем
перейти к Docker, разберём три основных формата, которые встречаются
повсеместно в Linux и DevOps.

### INI

Формат INI знаком вам по `php.ini` из главы 11. Простая структура:
секции в квадратных скобках, параметры в виде `ключ = значение`:

```ini
[server]
host = 127.0.0.1
port = 8080
debug = true

[database]
host = localhost
name = wordpress
```

Где встречается: `php.ini`, `my.cnf` (MariaDB), `~/.gitconfig`,
файлы systemd (unit-файлы из главы 09 похожи на INI).

### JSON

**JSON** (JavaScript Object Notation) — строгий формат с фигурными
скобками, кавычками и запятыми:

```json
{
  "server": {
    "host": "127.0.0.1",
    "port": 8080,
    "debug": true
  },
  "database": {
    "host": "localhost",
    "name": "wordpress"
  }
}
```

Где встречается: API веб-сервисов, конфигурации VS Code и npm,
файл `wp-config.json` (в новых версиях WordPress).

Особенности: не поддерживает комментарии, строгий синтаксис
(забыли запятую — ошибка).

### YAML

**YAML** (YAML Ain't Markup Language) — формат, который читается
почти как обычный текст. Структура задаётся **отступами**:

```yaml
server:
  host: 127.0.0.1
  port: 8080
  debug: true

database:
  host: localhost
  name: wordpress
```

Где встречается: Docker Compose (`docker-compose.yml`),
CI/CD конфиги (`.gitlab-ci.yml`, GitHub Actions), Ansible, Kubernetes.

**Правила YAML, которые нужно запомнить:**

- Отступы — **только пробелы**, никогда табуляция. Обычно 2 пробела.
- Вложенность определяется уровнем отступа.
- Списки обозначаются дефисом:

```yaml
fruits:
  - apple
  - banana
  - cherry
```

- Строки можно писать без кавычек (кроме специальных случаев).
- Комментарии начинаются с `#`.

> **Важно.** YAML очень чувствителен к отступам. Лишний или
> недостающий пробел приведёт к ошибке. Используйте редактор
> с подсветкой синтаксиса и будьте аккуратны.

### Сравнение форматов

| Формат | Синтаксис | Комментарии | Где чаще всего |
|--------|-----------|-------------|----------------|
| INI | `ключ = значение`, секции `[...]` | `;` или `#` | Конфиги приложений (php.ini, my.cnf) |
| JSON | Фигурные скобки, кавычки | Нет | API, веб-приложения |
| YAML | Отступы, двоеточия | `#` | DevOps (Docker Compose, CI/CD, Ansible) |

В следующих главах мы будем активно работать с YAML — для описания
Docker Compose и CI/CD конфигураций.

## Установка Docker

Docker не входит в стандартные репозитории Debian — мы установим его
из официального репозитория Docker. В главе 07 мы уже добавляли
сторонние репозитории — здесь принцип тот же.

### Подготовка

Установим пакеты, необходимые для работы с HTTPS-репозиториями:

```bash
sudo apt update
sudo apt install ca-certificates curl gnupg
```

### Добавление репозитория Docker

Скачаем и сохраним GPG-ключ Docker (он нужен для проверки
подлинности пакетов):

```bash
sudo install -m 0755 -d /etc/apt/keyrings
curl -fsSL https://download.docker.com/linux/debian/gpg \
  | sudo gpg --dearmor -o /etc/apt/keyrings/docker.gpg
sudo chmod a+r /etc/apt/keyrings/docker.gpg
```

Добавим репозиторий Docker в список источников:

```bash
echo \
  "deb [arch=$(dpkg --print-architecture) \
  signed-by=/etc/apt/keyrings/docker.gpg] \
  https://download.docker.com/linux/debian \
  $(. /etc/os-release && echo "$VERSION_CODENAME") stable" \
  | sudo tee /etc/apt/sources.list.d/docker.list > /dev/null
```

### Установка

```bash
sudo apt update
sudo apt install docker-ce docker-ce-cli containerd.io docker-buildx-plugin
```

### Настройка доступа без sudo

По умолчанию Docker требует прав root. Чтобы не писать `sudo` перед
каждой командой, добавьте своего пользователя в группу `docker`:

```bash
sudo usermod -aG docker $USER
```

Выйдите из сессии SSH и подключитесь заново, чтобы изменения
вступили в силу. Можно также выполнить `newgrp docker` в текущей
сессии.

### Проверка

```bash
docker run hello-world
```

```
Hello from Docker!
This message shows that your installation appears to be working correctly.
...
```

Если вы видите это сообщение — Docker установлен и работает.
Что произошло:

1. Docker не нашёл образ `hello-world` локально.
2. Скачал его из **Docker Hub** (реестр образов).
3. Создал контейнер из этого образа.
4. Контейнер вывел сообщение и завершился.

## Образы и контейнеры

Два ключевых понятия Docker:

- **Образ** (image) — шаблон для создания контейнера. Содержит
  файловую систему с приложением и всеми зависимостями. Образ
  неизменяем — его нельзя отредактировать, можно только создать
  новый.
- **Контейнер** (container) — запущенный экземпляр образа. Контейнер
  можно запускать, останавливать, удалять. Изменения внутри контейнера
  не влияют на образ.

Аналогия: образ — это рецепт, контейнер — приготовленное по рецепту
блюдо. Из одного образа можно создать сколько угодно контейнеров.

### Docker Hub

**Docker Hub** (hub.docker.com) — публичный реестр образов. Здесь
хранятся готовые образы для большинства популярных приложений:
nginx, PHP, MariaDB, WordPress, Ubuntu, Debian и тысячи других.

Образы бывают:

- **Официальные** — поддерживаются разработчиками приложений
  или Docker. Имя без префикса: `nginx`, `php`, `mariadb`.
- **Пользовательские** — созданы сообществом. Имя с префиксом:
  `username/myapp`.

Каждый образ имеет **теги** — версии:

| Образ | Описание |
|-------|----------|
| `nginx:latest` | Последняя стабильная версия (по умолчанию) |
| `nginx:1.27` | Конкретная версия 1.27 |
| `nginx:1.27-alpine` | Версия 1.27 на базе Alpine Linux (минимальный размер) |

> **Совет.** В продакшене всегда указывайте конкретную версию
> (`nginx:1.27`), а не `latest`. Иначе при обновлении образа
> может прийти версия с несовместимыми изменениями.

### Основные команды для работы с образами и контейнерами

Скачаем образ nginx:

```bash
docker pull nginx
```

```
Using default tag: latest
latest: Pulling from library/nginx
...
Status: Downloaded newer image for nginx:latest
```

Посмотрим список скачанных образов:

```bash
docker images
```

```
REPOSITORY    TAG       IMAGE ID       CREATED        SIZE
nginx         latest    a8758716bb6a   2 weeks ago    187MB
hello-world   latest    d2c94e258dcb   10 months ago  13.3kB
```

Запустим контейнер:

```bash
docker run -d --name test-nginx nginx
```

Флаг `-d` (detach) запускает контейнер в фоновом режиме,
`--name test-nginx` задаёт имя.

Посмотрим запущенные контейнеры:

```bash
docker ps
```

```
CONTAINER ID   IMAGE   COMMAND                  CREATED         STATUS         PORTS     NAMES
f3a1b2c3d4e5   nginx   "/docker-entrypoint.…"   5 seconds ago   Up 4 seconds   80/tcp    test-nginx
```

Посмотрим **все** контейнеры (включая остановленные):

```bash
docker ps -a
```

Остановим и удалим контейнер:

```bash
docker stop test-nginx
docker rm test-nginx
```

Удалим ненужный образ:

```bash
docker rmi hello-world
```

### Справочник команд

| Команда | Описание |
|---------|----------|
| `docker pull <образ>` | Скачать образ из Docker Hub |
| `docker images` | Список скачанных образов |
| `docker run <образ>` | Создать и запустить контейнер |
| `docker ps` | Список запущенных контейнеров |
| `docker ps -a` | Список всех контейнеров |
| `docker stop <имя>` | Остановить контейнер |
| `docker rm <имя>` | Удалить контейнер |
| `docker rmi <образ>` | Удалить образ |

## Запуск nginx в контейнере

Теперь запустим nginx так, чтобы он был доступен извне и
отдавал файлы нашего проекта.

### Проброс порта

Контейнер по умолчанию изолирован от хост-системы. Чтобы
обратиться к nginx внутри контейнера, нужно **пробросить порт** —
связать порт хоста с портом контейнера:

```bash
docker run -d --name my-nginx -p 8080:80 nginx
```

Флаг `-p 8080:80` означает: запросы на порт **8080 хоста**
перенаправлять на порт **80 контейнера** (стандартный порт nginx).

```
         Хост                    Контейнер
    ┌────────────┐          ┌────────────────┐
    │ порт 8080  │ ───────► │   порт 80      │
    │            │          │   (nginx)      │
    └────────────┘          └────────────────┘
```

Проверим:

```bash
curl http://localhost:8080
```

Вы увидите стандартную страницу «Welcome to nginx!». Nginx работает
внутри контейнера, но доступен через порт 8080 хоста.

Остановим контейнер перед следующим шагом:

```bash
docker stop my-nginx
docker rm my-nginx
```

### Монтирование файлов (volumes)

Стандартная страница nginx — не то, что нам нужно. Мы хотим отдавать
файлы нашего проекта. Для этого используем **тома** (volumes) —
механизм монтирования каталогов хоста внутрь контейнера.

Флаг `-v` связывает путь на хосте с путём в контейнере:

```bash
docker run -d --name my-nginx \
  -p 8080:80 \
  -v ~/wordpress-project/site:/usr/share/nginx/html:ro \
  nginx
```

Разберём `-v ~/wordpress-project/site:/usr/share/nginx/html:ro`:

| Часть | Значение |
|-------|----------|
| `~/wordpress-project/site` | Путь на хосте (наши файлы) |
| `:` | Разделитель |
| `/usr/share/nginx/html` | Путь внутри контейнера (куда nginx ищет файлы) |
| `:ro` | Read-only — контейнер может только читать эти файлы |

Проверим:

```bash
curl http://localhost:8080
```

Теперь nginx отдаёт наш `index.html` из `wordpress-project/site/`.

Можно смонтировать и конфигурацию nginx:

```bash
docker stop my-nginx
docker rm my-nginx

docker run -d --name my-nginx \
  -p 8080:80 \
  -v ~/wordpress-project/site:/usr/share/nginx/html:ro \
  -v ~/wordpress-project/config/nginx.conf:/etc/nginx/conf.d/default.conf:ro \
  nginx
```

Теперь контейнер использует наш конфиг и наши файлы — но сам nginx
установлен не на хосте, а внутри контейнера.

## Dockerfile: собственный образ

Команда `docker run` с множеством флагов `-v` неудобна — легко
забыть какой-нибудь параметр. **Dockerfile** позволяет описать образ
в файле: какой базовый образ использовать, какие файлы скопировать
внутрь, какие порты открыть.

### Создание Dockerfile

Остановим и удалим предыдущий контейнер:

```bash
docker stop my-nginx
docker rm my-nginx
```

Создадим файл `Dockerfile` в корне проекта:

```bash
cd ~/wordpress-project
nano Dockerfile
```

```dockerfile
FROM nginx:1.27

COPY config/nginx.conf /etc/nginx/conf.d/default.conf
COPY site/ /usr/share/nginx/html/

EXPOSE 80
```

Разберём каждую инструкцию:

| Инструкция | Описание |
|------------|----------|
| `FROM nginx:1.27` | Базовый образ — nginx версии 1.27 |
| `COPY config/nginx.conf ...` | Копирует наш конфиг внутрь образа |
| `COPY site/ ...` | Копирует файлы сайта внутрь образа |
| `EXPOSE 80` | Документирует, что контейнер слушает порт 80 |

> **Примечание.** `EXPOSE` не пробрасывает порт автоматически — это
> лишь документация для того, кто будет использовать образ. Проброс
> по-прежнему делается флагом `-p` при запуске.

### Сборка образа

```bash
docker build -t wordpress-nginx .
```

Флаг `-t wordpress-nginx` задаёт имя (тег) образа. Точка (`.`)
означает, что контекст сборки — текущий каталог (Docker отправит
все файлы из него в процесс сборки).

```
[+] Building 2.1s (8/8) FINISHED
 => [1/3] FROM nginx:1.27
 => [2/3] COPY config/nginx.conf /etc/nginx/conf.d/default.conf
 => [3/3] COPY site/ /usr/share/nginx/html/
 => exporting to image
 => => naming to docker.io/library/wordpress-nginx
```

Проверим, что образ создан:

```bash
docker images
```

```
REPOSITORY        TAG       IMAGE ID       CREATED          SIZE
wordpress-nginx   latest    b1c2d3e4f5a6   10 seconds ago   187MB
nginx             1.27      a8758716bb6a   2 weeks ago      187MB
```

### Запуск из собственного образа

```bash
docker run -d --name wp-nginx -p 8080:80 wordpress-nginx
```

Теперь не нужны флаги `-v` — конфиг и файлы сайта уже внутри образа.
Проверим:

```bash
curl http://localhost:8080
```

### .dockerignore

При сборке Docker отправляет всё содержимое текущего каталога
в процесс сборки. Чтобы исключить ненужные файлы (логи, каталог
`.git`, архивы), создайте файл `.dockerignore`:

```bash
nano .dockerignore
```

```
logs/
.git/
*.tar.gz
*.swp
```

Синтаксис такой же, как у `.gitignore`. После создания
`.dockerignore` пересоберите образ — он может стать меньше,
а сборка быстрее.

### Справочник инструкций Dockerfile

| Инструкция | Описание |
|------------|----------|
| `FROM <образ>` | Базовый образ |
| `COPY <src> <dst>` | Копировать файлы из контекста в образ |
| `RUN <команда>` | Выполнить команду при сборке (установить пакеты и т.п.) |
| `EXPOSE <порт>` | Документировать порт |
| `CMD ["..."]` | Команда по умолчанию при запуске контейнера |
| `WORKDIR <путь>` | Рабочий каталог внутри контейнера |
| `ENV <KEY>=<value>` | Установить переменную окружения |

## Управление контейнерами

### Жизненный цикл контейнера

```
docker run ──► Работает ──► docker stop ──► Остановлен
                                                │
               docker start ◄───────────────────┘
                    │
                    ▼
               Работает ──► docker rm ──► Удалён
```

Контейнер можно остановить и запустить снова — данные внутри
сохраняются, пока контейнер не удалён.

### Просмотр логов

```bash
docker logs wp-nginx
```

Выводит stdout и stderr контейнера. Для nginx это access.log
и error.log.

Следить за логами в реальном времени (как `journalctl -f`):

```bash
docker logs -f wp-nginx
```

Нажмите `Ctrl+C` для выхода.

### Выполнение команд внутри контейнера

Иногда нужно заглянуть «внутрь» контейнера — посмотреть файлы,
проверить конфигурацию. Для этого используется `docker exec`:

```bash
docker exec -it wp-nginx bash
```

Флаги `-it` обеспечивают интерактивный режим с терминалом. Вы
окажетесь внутри контейнера:

```
root@f3a1b2c3d4e5:/# ls /usr/share/nginx/html/
index.html
root@f3a1b2c3d4e5:/# cat /etc/nginx/conf.d/default.conf
...
root@f3a1b2c3d4e5:/# exit
```

Обратите внимание: внутри контейнера минимальная среда — многих
привычных утилит (vim, less, ps) может не быть. Это нормально:
контейнеры намеренно делают компактными.

Можно выполнить одну команду без входа в интерактивный режим:

```bash
docker exec wp-nginx nginx -t
```

```
nginx: the configuration file /etc/nginx/nginx.conf syntax is ok
nginx: configuration file /etc/nginx/nginx.conf test is successful
```

### Перезапуск контейнера

```bash
docker restart wp-nginx
```

Полезно после изменения конфигурации (если файлы смонтированы
через volume).

### Информация о контейнере

```bash
docker inspect wp-nginx
```

Выводит подробную информацию в формате JSON: сетевые настройки,
смонтированные тома, переменные окружения и многое другое. Можно
извлечь конкретное значение:

```bash
docker inspect -f '{{.NetworkSettings.IPAddress}}' wp-nginx
```

### Справочник команд управления

| Команда | Описание |
|---------|----------|
| `docker start <имя>` | Запустить остановленный контейнер |
| `docker stop <имя>` | Остановить контейнер |
| `docker restart <имя>` | Перезапустить контейнер |
| `docker logs <имя>` | Показать логи контейнера |
| `docker logs -f <имя>` | Следить за логами в реальном времени |
| `docker exec -it <имя> bash` | Войти внутрь контейнера |
| `docker exec <имя> <команда>` | Выполнить команду в контейнере |
| `docker inspect <имя>` | Подробная информация о контейнере |

### Очистка

Со временем накапливаются остановленные контейнеры и неиспользуемые
образы. Полезные команды для очистки:

```bash
# Удалить все остановленные контейнеры
docker container prune

# Удалить неиспользуемые образы
docker image prune

# Удалить всё неиспользуемое (контейнеры, образы, сети)
docker system prune
```

Docker запросит подтверждение перед удалением.

## Практические задания

1. Установите Docker из официального репозитория. Добавьте своего
   пользователя в группу `docker`. Проверьте установку командой
   `docker run hello-world`.

2. Скачайте образ nginx (`docker pull nginx`). Запустите контейнер
   с пробросом порта 8080 → 80. Откройте `http://localhost:8080`
   через `curl` и убедитесь, что видите стандартную страницу nginx.

3. Запустите nginx с монтированием каталога
   `~/wordpress-project/site` как корня веб-сервера. Проверьте,
   что `curl` показывает ваш `index.html`.

4. Добавьте монтирование конфигурации nginx из
   `~/wordpress-project/config/nginx.conf`. Проверьте, что контейнер
   использует ваш конфиг (`docker exec <имя> nginx -t`).

5. Создайте `Dockerfile` в `~/wordpress-project`. Соберите образ
   `wordpress-nginx` и запустите контейнер из него. Убедитесь, что
   сайт работает без флагов `-v`.

6. Создайте `.dockerignore`, исключающий `logs/`, `.git/`, `*.tar.gz`
   и `*.swp`. Пересоберите образ.

7. Посмотрите логи работающего контейнера (`docker logs`).
   Сделайте несколько запросов через `curl` и убедитесь, что они
   появляются в логах.

8. Войдите внутрь контейнера (`docker exec -it ... bash`).
   Посмотрите содержимое `/usr/share/nginx/html/` и
   `/etc/nginx/conf.d/`. Выйдите командой `exit`.

9. Остановите и удалите все контейнеры. Удалите образ `hello-world`.
   Выполните `docker system prune` для полной очистки.
