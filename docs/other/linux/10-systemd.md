# Этап 9. Процессы и сервисы (systemd)

## Что такое процесс

В Windows вы наверняка открывали «Диспетчер задач» (Ctrl+Shift+Esc)
и видели список запущенных программ: браузер, антивирус, проводник.
Каждая запущенная программа — это **процесс**.

В Linux всё точно так же. Каждая работающая программа — это процесс
с уникальным номером **PID** (Process ID). Когда вы подключаетесь
по SSH, запускается процесс оболочки bash. Когда набираете `ls`,
создаётся ещё один процесс, который выполняет эту команду и завершается.

Процессы образуют **дерево**: каждый процесс (кроме самого первого)
создан другим процессом — **родительским**. Самый первый процесс
в системе имеет PID 1 — это **systemd**, о котором мы подробно
поговорим ниже.

```
systemd (PID 1)
├── sshd
│   └── sshd (ваше подключение)
│       └── bash
│           └── ls
├── cron
└── nginx
    ├── nginx (worker)
    └── nginx (worker)
```

> **Привязка к проекту.** В этой главе мы научимся управлять
> процессами и сервисами. В практической части обернём наш скрипт
> `backup.sh` из главы 06 в systemd-сервис с таймером — получится
> автоматический бэкап wordpress-project по расписанию.

## Просмотр процессов

### ps — снимок процессов

Команда `ps` показывает список процессов **в данный момент** (как
фотография — снимок, а не видео):

```bash
ps aux
```

| Флаг | Значение |
|------|----------|
| `a` | Показать процессы всех пользователей |
| `u` | Показать подробный формат (пользователь, CPU, память) |
| `x` | Показать процессы без терминала (демоны, сервисы) |

Вывод содержит много строк. Чтобы найти конкретный процесс,
используйте `grep` (глава 05):

```bash
ps aux | grep sshd
```

```
root       654  0.0  0.3  15432  6892 ?        Ss   10:00   0:00 sshd: /usr/sbin/sshd -D
root       987  0.0  0.4  16784  8256 ?        Ss   10:05   0:00 sshd: student [priv]
student    991  0.0  0.2  16784  5432 ?        S    10:05   0:00 sshd: student@pts/0
student   1023  0.0  0.0   6420   720 pts/0    S+   10:10   0:00 grep sshd
```

Обратите внимание: последняя строка — это сам процесс `grep`,
который тоже содержит слово «sshd». Это нормально.

Основные столбцы вывода `ps aux`:

| Столбец | Значение |
|---------|----------|
| `USER` | Владелец процесса |
| `PID` | Идентификатор процесса |
| `%CPU` | Использование процессора |
| `%MEM` | Использование памяти |
| `COMMAND` | Команда, которой запущен процесс |

### pstree — дерево процессов

Чтобы увидеть иерархию процессов (кто кого породил):

```bash
pstree
```

С отображением PID:

```bash
pstree -p
```

### top — интерактивный мониторинг

Команда `top` показывает процессы в реальном времени — как «Диспетчер
задач» в Windows. Информация обновляется каждые несколько секунд:

```bash
top
```

Полезные клавиши внутри `top`:

| Клавиша | Действие |
|---------|----------|
| `q` | Выйти |
| `k` | Завершить процесс (спросит PID) |
| `M` | Сортировать по памяти |
| `P` | Сортировать по CPU |
| `1` | Показать все ядра процессора |

> **htop.** Существует улучшенная версия — `htop`, с цветным
> интерфейсом и более удобным управлением. Если хотите попробовать,
> установите её: `sudo apt install htop`. Для нашего курса достаточно
> стандартного `top`.

## Управление процессами

### Сигналы

Чтобы повлиять на работающий процесс (остановить, перезапустить),
ему отправляют **сигнал**. Самые важные:

| Сигнал | Номер | Описание |
|--------|-------|----------|
| `SIGTERM` | 15 | Вежливая просьба завершиться. Процесс может сохранить данные и выйти корректно |
| `SIGKILL` | 9 | Принудительное завершение. Процесс не может его проигнорировать |
| `SIGHUP` | 1 | «Перечитай конфигурацию». Многие сервисы перезагружают настройки без перезапуска |

### kill — отправка сигнала

```bash
kill 1234          # отправить SIGTERM процессу с PID 1234
kill -9 1234       # отправить SIGKILL (принудительное завершение)
kill -HUP 1234     # отправить SIGHUP (перечитать конфигурацию)
```

Всегда начинайте с `kill` (SIGTERM) — дайте процессу завершиться
корректно. Используйте `kill -9` только если процесс не реагирует
на обычный `kill`.

### Фоновые процессы

Когда вы запускаете команду в терминале, она занимает его до
завершения. Но можно запустить команду **в фоне**:

```bash
sleep 60 &
```

Символ `&` в конце запускает команду в фоновом режиме. Терминал
остаётся свободным.

Полезные клавиши и команды:

| Действие | Команда |
|----------|---------|
| Прервать текущий процесс | `Ctrl+C` (отправляет SIGINT) |
| Приостановить текущий процесс | `Ctrl+Z` (отправляет SIGTSTP) |
| Посмотреть фоновые задачи | `jobs` |
| Вернуть задачу на передний план | `fg` |
| Продолжить задачу в фоне | `bg` |

Пример работы:

```bash
sleep 120          # запустили, терминал занят
                   # нажимаем Ctrl+Z — процесс приостановлен
jobs               # видим: [1]+  Stopped  sleep 120
bg                 # продолжить в фоне
jobs               # видим: [1]+  Running  sleep 120 &
fg                 # вернуть на передний план
                   # нажимаем Ctrl+C — процесс завершён
```

## Что такое systemd

До сих пор мы работали с обычными процессами — запустили команду,
она поработала и завершилась. Но в системе есть программы, которые
должны работать постоянно: SSH-сервер, веб-сервер, планировщик задач.
Такие постоянно работающие программы называют **сервисами** (или
**демонами**).

Кто-то должен запустить все сервисы при загрузке системы, следить
за ними и перезапускать, если они упали. Эту роль выполняет **systemd** —
система инициализации и управления сервисами в Debian (и большинстве
современных дистрибутивов Linux).

systemd — это тот самый процесс с PID 1, корень дерева процессов.
Он запускается ядром первым и затем поднимает всю остальную систему:
монтирует диски, настраивает сеть, запускает сервисы.

Каждый объект, которым управляет systemd, называется **юнитом** (unit).
Юниты бывают разных типов:

| Тип | Расширение | Описание |
|-----|-----------|----------|
| Сервис | `.service` | Программа-демон (nginx, sshd, cron) |
| Таймер | `.timer` | Запуск задач по расписанию (замена cron) |
| Сокет | `.socket` | Активация сервиса при подключении к порту |
| Точка монтирования | `.mount` | Монтирование файловой системы |
| Цель | `.target` | Группа юнитов (аналог «уровня запуска») |

Чаще всего вы будете работать с типами `.service` и `.timer`.

## systemctl: управление сервисами

`systemctl` — главная команда для работы с systemd. Рассмотрим
её на примере сервиса SSH, который уже работает в вашей системе
(без него вы бы не смогли подключиться).

### Статус сервиса

```bash
systemctl status ssh
```

```
● ssh.service - OpenBSD Secure Shell server
     Loaded: loaded (/lib/systemd/system/ssh.service; enabled; preset: enabled)
     Active: active (running) since Mon 2026-03-16 10:00:00 MSK; 2h ago
       Docs: man:sshd(8)
   Main PID: 654 (sshd)
      Tasks: 1 (limit: 4647)
     Memory: 5.2M
        CPU: 132ms
     CGroup: /system.slice/ssh.service
             └─654 "sshd: /usr/sbin/sshd -D [listener] 0 of 10-100 startups"
```

Ключевые строки:

| Строка | Значение |
|--------|----------|
| `Loaded: loaded (...; enabled)` | Юнит загружен; **enabled** — запускается автоматически при загрузке |
| `Active: active (running)` | Сервис работает прямо сейчас |
| `Main PID: 654` | PID основного процесса |

### Запуск, остановка, перезапуск

```bash
sudo systemctl start ssh       # запустить сервис
sudo systemctl stop ssh        # остановить сервис
sudo systemctl restart ssh     # перезапустить (stop + start)
sudo systemctl reload ssh      # перечитать конфигурацию без перезапуска
```

> **restart vs reload.** `restart` полностью останавливает и запускает
> сервис заново — клиенты на секунду теряют связь. `reload` просит
> сервис перечитать конфигурацию «на лету», без остановки. Не все
> сервисы поддерживают `reload` — если не поддерживает, команда
> вернёт ошибку.

### Автозапуск при загрузке

```bash
sudo systemctl enable ssh      # включить автозапуск
sudo systemctl disable ssh     # отключить автозапуск
```

`enable` не запускает сервис прямо сейчас — только добавляет его
в автозагрузку. Чтобы сделать и то, и другое одной командой:

```bash
sudo systemctl enable --now ssh
```

### Список сервисов

Посмотреть все загруженные сервисы:

```bash
systemctl list-units --type=service
```

Только активные (работающие):

```bash
systemctl list-units --type=service --state=running
```

## journalctl: чтение логов

systemd ведёт централизованный журнал — все сообщения от всех сервисов
собираются в одном месте. Читать этот журнал можно командой `journalctl`.

### Логи конкретного сервиса

```bash
journalctl -u ssh
```

Флаг `-u` (unit) фильтрует записи по имени юнита.

### Последние записи

```bash
journalctl -u ssh -n 20        # последние 20 строк
```

### Следить в реальном времени

```bash
journalctl -u ssh -f
```

Флаг `-f` (follow) работает как `tail -f` — показывает новые записи
по мере их появления. Выход — `Ctrl+C`.

### Фильтрация по времени

```bash
journalctl -u ssh --since "2026-03-16 10:00"
journalctl -u ssh --since "1 hour ago"
journalctl -u ssh --since today
```

> **Подробнее о логировании.** В этой главе мы рассматриваем
> `journalctl` как инструмент диагностики сервисов. Полноценный
> разбор логирования и мониторинга — в главе 12.

## Устройство unit-файла

Мы научились управлять сервисами снаружи. Теперь заглянем внутрь —
посмотрим, как описан сервис.

### Где хранятся unit-файлы

Unit-файлы лежат в нескольких каталогах:

| Каталог | Назначение |
|---------|-----------|
| `/lib/systemd/system/` | Файлы, установленные пакетами (nginx, ssh, cron) |
| `/etc/systemd/system/` | Файлы, созданные администратором (имеют приоритет) |

Посмотреть содержимое unit-файла можно командой:

```bash
systemctl cat ssh
```

### Разбор ssh.service

Рассмотрим упрощённую структуру unit-файла SSH-сервера:

```ini
[Unit]
Description=OpenBSD Secure Shell server
Documentation=man:sshd(8)
After=network.target

[Service]
Type=notify
ExecStartPre=/usr/sbin/sshd -t
ExecStart=/usr/sbin/sshd -D
ExecReload=/bin/kill -HUP $MAINPID
Restart=on-failure

[Install]
WantedBy=multi-user.target
```

Разберём по секциям:

**Секция [Unit]** — общее описание:

| Директива | Значение |
|-----------|----------|
| `Description` | Человекочитаемое описание сервиса |
| `Documentation` | Ссылка на документацию |
| `After` | Запускать после указанных юнитов (здесь — после настройки сети) |

**Секция [Service]** — как запускать и контролировать:

| Директива | Значение |
|-----------|----------|
| `Type` | Тип сервиса (`notify`, `simple`, `oneshot`, `forking`) |
| `ExecStartPre` | Команда перед запуском (здесь — проверка конфигурации) |
| `ExecStart` | Основная команда запуска |
| `ExecReload` | Команда для `systemctl reload` (перечитать конфигурацию) |
| `Restart` | Когда перезапускать (`on-failure` — только при сбое) |

**Секция [Install]** — автозапуск:

| Директива | Значение |
|-----------|----------|
| `WantedBy` | К какой цели (target) привязать при `systemctl enable` |

> **Что такое multi-user.target?** Это цель, которая соответствует
> обычному режиму работы сервера (многопользовательский режим с сетью,
> но без графического интерфейса). Большинство серверных сервисов
> привязываются именно к ней.

## Практика: оборачиваем backup.sh в systemd-сервис

В главе 06 мы написали скрипт `backup.sh`, который создаёт архив
каталога `wordpress-project`. Сейчас мы превратим его в systemd-сервис
с таймером — чтобы бэкап запускался автоматически по расписанию.
Это современная альтернатива планировщику `cron`.

### Шаг 1. Подготавливаем скрипт

Убедимся, что скрипт находится в правильном месте и имеет права
на выполнение:

```bash
ls -l ~/wordpress-project/backup.sh
```

Если прав на выполнение нет, добавим:

```bash
chmod +x ~/wordpress-project/backup.sh
```

### Шаг 2. Создаём service-файл

Создадим файл `/etc/systemd/system/wordpress-backup.service`:

```bash
sudo nano /etc/systemd/system/wordpress-backup.service
```

Содержимое:

```ini
[Unit]
Description=Backup wordpress-project
After=local-fs.target

[Service]
Type=oneshot
User=webmaster
ExecStart=/home/student/wordpress-project/backup.sh
```

Разберём, что здесь написано:

- `Type=oneshot` — сервис выполняет задачу один раз и завершается
  (в отличие от `simple`, который работает постоянно).
- `User=webmaster` — скрипт запускается от имени пользователя
  `webmaster`, которого мы создали в главе 08.
- `ExecStart` — путь к нашему скрипту бэкапа.

### Шаг 3. Создаём timer-файл

Создадим файл `/etc/systemd/system/wordpress-backup.timer`:

```bash
sudo nano /etc/systemd/system/wordpress-backup.timer
```

Содержимое:

```ini
[Unit]
Description=Run wordpress-backup daily

[Timer]
OnCalendar=daily
Persistent=true

[Install]
WantedBy=timers.target
```

Разберём секцию `[Timer]`:

| Директива | Значение |
|-----------|----------|
| `OnCalendar=daily` | Запускать раз в сутки (в полночь) |
| `Persistent=true` | Если система была выключена в момент запуска — выполнить при следующем включении |

Другие варианты расписания `OnCalendar`:

| Значение | Когда |
|----------|-------|
| `hourly` | Каждый час |
| `daily` | Раз в сутки |
| `weekly` | Раз в неделю |
| `*-*-* 03:00:00` | Каждый день в 3:00 |
| `Mon *-*-* 09:00:00` | Каждый понедельник в 9:00 |

### Шаг 4. Запускаем таймер

Сообщим systemd, что появились новые unit-файлы:

```bash
sudo systemctl daemon-reload
```

Включим и запустим таймер:

```bash
sudo systemctl enable --now wordpress-backup.timer
```

### Шаг 5. Проверяем

Посмотрим, что таймер активен:

```bash
systemctl list-timers
```

В списке должен появиться `wordpress-backup.timer` с указанием
следующего времени срабатывания.

Запустим сервис вручную, чтобы убедиться, что он работает:

```bash
sudo systemctl start wordpress-backup.service
```

Проверим статус:

```bash
systemctl status wordpress-backup.service
```

Посмотрим лог:

```bash
journalctl -u wordpress-backup.service
```

Вы должны увидеть сообщения о выполнении скрипта и успешном
завершении.

## Справочник команд

### Процессы

| Команда | Описание |
|---------|----------|
| `ps aux` | Показать все процессы |

### Фоновые процессы

| Команда | Описание |
|---------|----------|
| `команда &` | Запустить в фоне |

| `Ctrl+Z` | Приостановить текущий процесс |

| `fg` | Вернуть задачу на передний план |
| `bg` | Продолжить задачу в фоне |

### systemctl

| Команда | Описание |
|---------|----------|
| `systemctl status имя` | Статус сервиса |
| `systemctl start имя` | Запустить сервис |
| `systemctl stop имя` | Остановить сервис |
| `systemctl restart имя` | Перезапустить сервис |
| `systemctl reload имя` | Перечитать конфигурацию |

|---------|----------|

| `journalctl -u имя -f` | Следить в реальном времени |
| `journalctl -u имя --since today` | Логи с начала дня |

## Практические задания

1. Выполните `ps aux` и найдите процесс вашей bash-сессии. Запишите
   его PID. Совпадает ли он с выводом переменной `echo $$`?

2. Выполните `pstree -p` и найдите свою сессию. Проследите цепочку
   от `systemd (1)` до вашего `bash`. Какие промежуточные процессы
   в этой цепочке?

3. Запустите `sleep 300 &` в фоне. Найдите этот процесс через
   `ps aux | grep sleep`. Завершите его командой `kill`. Убедитесь,
   что он исчез (`jobs`).

4. Выполните `systemctl status ssh`. Какой PID у основного процесса?
   Когда сервис был запущен? Включён ли автозапуск (enabled/disabled)?

5. Посмотрите unit-файл сервиса cron: `systemctl cat cron`. Найдите
   директиву `ExecStart`. Какая команда запускает планировщик?

6. Выполните `systemctl list-units --type=service --state=running`.
   Сколько сервисов сейчас работает в системе? Найдите знакомые
   (ssh, cron).

7. Посмотрите логи SSH за сегодня:
   `journalctl -u ssh --since today`. Видны ли в них ваши подключения?

8. Если вы создали таймер `wordpress-backup.timer`, проверьте его
   статус через `systemctl list-timers`. Когда запланирован следующий
   запуск? Запустите сервис вручную (`sudo systemctl start
   wordpress-backup.service`) и убедитесь в успешном выполнении
   через `journalctl`.
