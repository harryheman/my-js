# Ускорение загрузки следующей страницы

Один из самых эффективных способов улучшения производительности навигации -  инициализация загрузки следующей страницы, которую пользователь собирается посетить. Этого можно достичь с помощью API спекулятивных правил (Speculation Rules).

## Как это работает

Спекулятивная загрузка работает через спекулятивные правила в формате JSON, сообщающие браузеру, какие ссылки могут быть предварительно запрошены (prefetched) или отрендерены (prerendered), что ускоряет загрузку страницы, когда пользователь кликает по такой ссылке.

Правила могут быть списком URL, определяемым с помощью ключа `urls` (правило списка), или ключом `where`, содержащим набор `href` и селекторов CSS для поиска ссылок на странице (правило документа).

Правила могут также включать опциональное свойство `eagerness`, определяющее, когда страница должна быть предварительно запрошена или отрендерена. Свойство `eagerness` может быть установлено в значение `immediate`, `eager`, `moderate` или `conservative`. `immediate` означает "как можно раньше", другие ждут сигнала пользователя, такого как наведение курсора на ссылку на короткое время, более продолжительное время или клик на странице, соответственно.

Правила могут сочетаться с разными настройками загрузки для предварительного запроса страницы ии ее предварительного рендеринга при повышении активности пользователя.

## Когда использовать

Спекулятивная загрузка особенно полезна для статических страниц, контент которых редко меняется, и которые дешевле создавать, особенно если они кэшируются на границе. Она также может использоваться для динамичных страниц, но важно быть осторожным в отношении потенциально устаревшего контента.

Спекулятивная загрузка, как правило, используется для ссылок одного источника (same-origin), хотя более продвинутые настройки позволяют некоторые межсайтовые спекуляции. В этом руководстве предполагается один источник.

Более активная спекулятивная загрузка - хороший выбор для страниц, которые, вероятно, будут посещены следующими, например, заголовок статьи или следующая страница в поэтапном процессе, таком как прохождение курса.

Менее активная спекулятивная загрузка - хороший выбор, когда намерения пользователя не так очевидны, когда на странице много ссылок с равной вероятностью посещения. Ожидая больше сигналов, таких как наведение курсора или клик по странице, можно начать обработку следующей страницы и улучшить пользовательский опыт, даже если она не была полностью предварительно запрошена и отрендерена.

Аналогично, предварительный запрос - более консервативный выбор, чем предварительный рендеринг, который использует меньше ресурсов (как на клиенте, так и на сервере), но предоставляет меньше выгод пользователю. Это хороший выбор для начальной реализации, расширяющийся до предварительного рендеринга, когда разработчик явно его запрашивает.

## Как использовать

Спекулятивные правила определяются в формате JSON и включаются в тег `<script type="speculationrules">`. Правила могут содержаться в `<head>` или `<body>` документа или добавляться динамически с помощью JavaScript.

Также может быть использован `tag`, как на глобальном уровне, так и для отдельных правил. При установке, этот тег включается в заголовок `Sec-Speculation-Tags` и позволяет определить на сервере, какие спекуляции были выполнены.

### Пример простого правила списка для предварительного запроса предопределенных URL

```html
<script type="speculationrules">
  {
    "tag": "product-page-speculations",
    "prefetch": [
      {
        "urls": ["/product/1", "/product/2", "/product/3"]
      }
    ]
  }
</script>
```

### Пример простого правила документа для предварительного рендеринга всех ссылок одного источника на странице

```html
<script type="speculationrules">
  {
    "tag": "all-links-speculations",
    "prerender": [{
      "where": { "href_matches": "/*" }
    }]
  }
</script>
```

### Пример сложного правила документа для предварительного рендеринга ссылок с исключениями для интерактивных страниц

```html
<script type="speculationrules">
  {
    "tag": "speculations-with-exclusions",
    "prerender": [{
      "where": {
        "and": [
          { "href_matches": "/*" },
          { "not": {"href_matches": "/wp-admin"}},
          { "not": {"href_matches": "/*\\?*(^|&)add-to-cart=*"}},
          { "not": {"selector_matches": ".do-not-prerender"}},
          { "not": {"selector_matches": "[rel~=nofollow]"}}
        ]
      }
    }]
  }
</script>
```

### Пример смешанного набора правил

В этом примере показан набор правил, который активно предварительно запрашивает все ссылки (`eager`), и затем выполняет их предварительный рендеринг при получении дополнительных сигналов (`moderate`).

```html
<script type="speculationrules">
  {
    "prefetch": [{
      "tag": "prefetch-speculations",
      "where": { "href_matches": "/*" },
      "eagerness": "eager"
    }],
    "prerender": [{
      "tag": "prerender-speculations",
      "where": { "href_matches": "/*" },
      "eagerness": "moderate"
    }]
  }
</script>
```

## Лучшие практики

- Используйте спекулятивные правила для предварительного получения и рендеринга страниц, которые пользователь посетит следующими с большой долей вероятности.
- Используйте спекулятивные правила для статичных сайтов, когда контент меняется редко и когда страницы дешево создавать - особенно, если они кэшируются на границе.
- Проявляйте осторожность при использовании спекулятивных правил для динамичных страниц, когда контент меняется часто, может устареть, и когда страницы дороже создавать.
- Используйте правила документа, а не правила списка, поскольку они более гибкие, позволяют использовать одно и тоже правило на нескольких страницах и могут использоваться для предварительного получения и рендеринга заранее неизвестных страниц.
- Учитывайте компромиссы предварительного получения и рендеринга и выбирайте подходящий для вашего случая. Предварительный рендеринг дороже, чем предварительное получение, и может вызывать больше нежелательных побочных эффектов в сложных приложениях, отображающих динамичные данные, но обеспечивает лучший пользовательский опыт.
- Учитывайте компромиссы разных уровней `eagerness` и выбирайте подходящий для вашего случая. Более активная спекуляция обеспечивает лучший пользовательский опыт, но использует больше ресурсов и может вызывать больше нежелательных побочных эффектов в сложных приложениях, отображающих динамичные данные.
- Не злоупотребляйте правилами спекуляции, например, путем их применения к каждой ссылке на странице. Браузеры имеют ограничения (2 спекуляции для не активных спекуляций). `immediate` должно использоваться для небольшого количества ссылок.
- Не спекулируйте URL, которые запускают модификации состояния, например, `/logout` или `/add-to-cart`, и явно исключайте их из спекулятивных правил, если они подпадают под правила документа.
- Не используйте правила спекуляции в одностраничных приложениях (SPA). Спекулятивные правила предназначены для многостраничных приложений (MPA), когда браузер переходит к новому документу при каждой навигации. В SPA браузер этого не делает, поэтому правила спекуляции не будут работать ожидаемо.

## Поддержка и страховка

Спекулятивная загрузка - новая возможность, которая пока имеет ограниченную поддержку браузеров.

Однако, спекулятивная загрузка - это прогрессивное улучшение. Использовать ее совершенно безопасно, но она может существенно улучшить производительность. Если браузер не поддерживает спекулятивные правила, он будет просто их игнорировать.
