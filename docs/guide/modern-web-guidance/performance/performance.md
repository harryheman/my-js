# Руководство по производительности

## Оптимизация критичного пути рендеринга

Критичный путь рендеринга (Critical Rendering Path, CRP) определяет, как быстро браузер преобразует HTML, CSS и JavaScript в отрисованные пиксели.

### Хорошо

* Внедряйте критичный CSS прямо на страницу: извлеките стили, необходимые для видимого при загрузке страницы контента, и внедрите их прямо в `<head>`. Отложите загрузку остальной таблицы стилей.
* Используйте атрибуты `async` или `defer` для всех некритичных скриптов: не позволяйте JavaScript блокировать парсинг DOM. Используйте `defer` для скриптов, зависящих от DOM или друг от друга, а `async` - для независимых скриптов. `type="module"` - это предпочтительный вариант для современного JavaScript, который откладывается по умолчанию и не требует наличия `defer`, но может дополняться `async` для независимых модульных скриптов.
* Разделяйте CSS по медиа-запросам: используйте атрибут `media` на тегах `<link>`, чтобы браузер мог загружать неиспользуемые таблицы стилей (например, стили для печати или десктопные стили на мобильном телефоне) без блокировки рендеринга.
* Используйте подсказки ресурсов (resource hints): добавляйте `preconnect` или `dns-prefetch` для второстепенных сторонних доменов (например, хостинги шрифтов или конечные точки API) для установки ранних рукопожатий (early handshakes) TLS.

### Плохо

* Не используйте `@import` в CSS: это создает последовательные цепочки запросов, которые задерживают создание объектной модели CSS (CSSOM).
* Не размещайте большие некритичные скрипты в `<head>`: это задерживает создание DOM до их полной загрузки, парсинга и выполнения.
* Не загружайте невидимый или недостижимый CSS/JS: убедитесь, что инструменты сборки выполняют тряску дерева (tree-shaking) и минификацию CSS для удаления недостижимого кода перед развертыванием приложения.

### Пример кода

Отложенная загрузка некритичного CSS и JS:

```html
<!-- Внедряем критичные стили прямо в head -->
<style>
  body { margin: 0; font-family: system-ui; }
  .hero { min-height: 100vh; }
</style>

<!-- Откладываем некритичный CSS -->
<link rel="preload" href="non-critical.css" as="style" onload="this.onload=null;this.rel='stylesheet'">
<noscript><link rel="stylesheet" href="non-critical.css"></noscript>

<!-- Загружаем CSS условно, на основе области просмотра -->
<link rel="stylesheet" href="mobile.css" media="(max-width: 768px)">

<!-- Откладываем выполнение JavaScript -->
<script defer src="app-bundle.js"></script>
```

### Подсказки ресурсов

| Подсказка      | Случай использования                                                    | Примеры                                            |
|:---------------|:------------------------------------------------------------------------|:---------------------------------------------------|
| `preconnect`   | Разрешение TLS/DNS для известных сторонних API                          | Конечные точки API, сервисы шрифтов                |
| `dns-prefetch` | Отложенный запасной вариант для некритичных сторонних источников        | Реклама, аналитика                                 |
| `preload`      | Ресурсы из того же источника, которые требуются для текущего рендеринга | Основные изображения, блокирующие рендеринг шрифты |
| `prefetch`     | Ресурсы, необходимые для следующей страницы                             | Сборка следующей страницы, отображение деталей     |

Ментальная модель в одном предложении: "`preconnect` для доменов, `preload` для области просмотра, `prefetch` для будущего".

## Отрисовка наибольшего контента (LCP) и получение ресурсов

LCP измеряет время, необходимое для рендеринга самого большого видимого текста или изображения в области просмотра. Оптимизируйте LCP через приоритизацию видимых элементов и их предварительную подготовку.

### Хорошо

* Используйте `fetchpriority="high"` для изображения LCP: сигнализирует эвристическому движку браузера повысить приоритет изображения над скриптами и некритичными ресурсами.
* Определяйте изображение LCP в стандартном HTML: убедитесь, что тег `<img>` присутствует в сыром HTML ответа, чтобы сканер предварительной загрузки (preload scanner) "видел" его сразу. Не монтируйте элемент LCP с помощью JavaScript.
* Предварительно загружайте фоновые изображения, являющиеся LCP: если элемент LCP находится в CSS (`background-image`), ускорьте его раннее обнаружение с помощью `<link rel="preload" as="image">` в сочетании с `fetchpriority="high"`.
* Используйте `fetchpriority="low"` для понижения приоритета второстепенных элементов: понизьте приоритет больших изображений или каруселей, которые находятся за пределами экрана и не являются элементами LCP.

### Плохо

* Не загружайте лениво изображение LCP: никогда не применяйте `loading="lazy"` к видимым при загрузке страницы изображениям. Это задерживает получение данных до завершения расчета макета, что серьезно ухудшает LCP.
* Не злоупотребляйте `fetchpriority="high"`: приоритизация - это механизм с нулевой суммой. Повышение приоритета слишком многих ресурсов создает сетевые конфликты и нивелирует выгоды использования этого атрибута.
* Не реализуйте сложные загрузчики JavaScript для основного раздела: клиентский рендеринг элемента LCP приводит к цепочке последовательных запросов (HTML -> JS -> выполнение -> запрос изображения).

### Пример кода

Оптимизация изображения LCP:

```html
<!-- Стандартное изображение LCP -->
<img
  src="/images/hero.webp"
  alt="Hero Product"
  fetchpriority="high"
  decoding="sync"
  width="1200"
  height="600"
>

<!-- Предзагрузка основанного на CSS фона LCP -->
<link rel="preload" as="image" href="/images/bg-hero.webp" fetchpriority="high" type="image/webp">

<!-- Откладывание не LCP изображения карусели, находящегося за пределами экрана -->
<img src="/images/carousel-2.webp" fetchpriority="low" alt="Slide 2">
```

## Взаимодействие до следующей отрисовки (INP) и разблокировка основного потока

INP измеряет длительность всех интерактивных событий на протяжении всего жизненного цикла страницы. Основной причиной плохого INP являются долгие задачи JavaScript, блокирующие основной поток.

### Хорошо

* Разделяйте долгие задачи: любое выполнение JavaScript, превышающее 50 мс, должно разделяться. Периодически возвращайте управление браузеру, чтобы он мог обработать пользовательский ввод.
* Используйте `scheduler.yield()` с резервом: используйте современный API `scheduler.yield()` для помещения возобновления задачи в начало очереди с резервом `setTimeout`, обернутым в промис, для старых браузеров.
* Задерживайте выполнение частых обработчиков событий с помощью таких техник, как "debounce" или "throttle": огранивайте частоту выполнения обработчиков таких событий, как `scroll`, `resize` или частых событий `input`.
* Отделяйте обновления UI от тяжелых вычислений: обновляйте UI синхронно для предоставления незамедлительной обратной связи при фоновой обработке веб-воркером или отложенном выполнении задачи.

### Плохо

* Не полагайтесь на `setTimeout(..., 0)` для возобновления задач: стандартный `setTimeout` помещает возобновление в конец очереди, что может привести к большой задержке при наличии в очереди других задач.
* Не засоряйте макет: избегайте чередования операций чтения (`offsetHeight`, `getBoundingClientRect`) и записи (`style.height`) в DOM внутри одного цикла. Сначала выполняйте групповые (batch) чтения DOM, затем - записи в DOM.
* Не блокируйте поток рекуррентными таймерами: избегайте частого `setInterval`, насыщающего основной поток.

### Пример кода

`scheduler.yield` - полифилл и использование:

```javascript
// Полифилл для передачи управления основному потоку
async function yieldToMain() {
  if ('scheduler' in window && 'yield' in scheduler) {
    return await scheduler.yield();
  }
  return new Promise(resolve => setTimeout(resolve, 0));
}

// Обработка большого массива без блокировки пользовательского ввода
async function processLargeList(items) {
  for (let i = 0; i < items.length; i++) {
    processItem(items[i]);

    // Передаем управление основном потоку каждые 50 итераций для обработки рендеринга/действий пользователя
    if (i % 50 === 0) {
      await yieldToMain();
    }
  }
}
```

### Разделение задач основного потока

Правило 50 мс для INP:

- `< 50 мс`: выполняем синхронно;
- `50 - 250 мс`: разделяем задачу и передаем управление основному потоку с помощью `scheduler.yield()`.
- `> 250 мс`: делегируем задачу веб-воркеру.

## Управление сторонними скриптами

Сторонние скрипты (аналитика, реклама, виджеты чатов) - основные источники перегрузки основного потока.

### Хорошо

* Не позволяйте сторонним скриптам блокировать основной контент: используйте `defer` для всех некритичных сторонних скриптов и загружайте их в подвале страницы, в не в `<head>`.
* Размещайте критически важные сторонние зависимости у себя: сократите количество запросов DNS и внедрите собственную логику управления кэшем, разместив сторонние библиотеки в своем домене.

### Пример кода

Выполнение сторонних скриптов:

```html
<!-- Размещайте сторонние скрипты в конце страницы с атрибутом defer -->
<script defer src="http://www.example.com/third-party.js"></script>
```

## Рендеринг CSS и оптимизация сдерживания

Рендеринг включает в себя вычисления макета (Layout), стилей (Style), отрисовки (Paint) и компоновки (Compositing). Сдерживание (containment) CSS ограничивает область этих вычислений, что полезно на больших сложных страницах, где такие вычисления могут вызывать проблемы производительности.

### Хорошо

* Используйте `content-visibility: auto` для разделов за пределами экрана на больших сложных страницах: позвольте браузеру пропускать макетирование и отрисовку целых поддеревьев, пока они не достигнут области просмотра.
* Используйте `content-visibility` вместе с `contain-intrinsic-size`: избегайте сдвигов макета и прыжков прокрутки путем предоставления заполнителя (placeholder) высоты/ширины для "пропущенных" контейнеров.
* Применяйте явное сдерживание CSS (`contain`): используйте `contain: layout style paint` для изолированных компонентов UI (вроде модалок или виджетов) для предотвращения перерисовок страницы, вызванных их внутренними изменениями.

### Плохо

* Не применяйте `content-visibility: auto` к небольшим простым страницам: выгоды будут незначительными, и существует риск побочных эффектов в виде прыжков контента.
* Не применяйте `content-visibility: auto` к контенту, видимому при загрузке страницы: браузер по-прежнему будет его оценивать, но принудительное пропускание через механизм сдерживания без необходимости добавляет небольшие накладные расходы для видимых элементов.
* Не злоупотребляйте глобальным `will-change`: неразборчивое применение `will-change: transform` к множеству элементов потребляет чрезмерное количество видеопамяти, что приводит к сбоям графического процессора или замедленному рендерингу.
* Не забывайте про доступность при скрытии элементов: `content-visibility: auto` сохраняет элементы в DOM для устройств чтения с экрана. Если контент должен быть скрыт от вспомогательных устройств, управляйте атрибутами `aria-hidden` вручную.

### Пример кода

Видимость контента и сдерживание CSS:

```css
/* Оптимизируем длинный список статей за пределами экрана */
.article-list-item {
  content-visibility: auto;
  contain-intrinsic-size: auto 600px; /* Предоставляем заполнитель шириной 600px */
}

/* Изолируем сложный виджет для предотвращения засорения макета */
.isolated-widget {
  contain: layout style paint;
}

/* Аппаратно ускоряем анимацию только при наведении курсора */
.interactive-button:hover {
  will-change: transform;
  transform: scale(1.05);
}
```

## Современные оптимизации изображений и другого медиаконтента

Изображения часто являются наибольшей полезной нагрузкой (payload) веб-страниц. Их оптимизация требует согласования формата, отзывчивого масштабирования и стабилизации макета.

### Хорошо

* Используйте современные форматы (AVIF / WebP): используйте элемент `<picture>` для предоставления AVIF (лучшее сжатие), откатываясь сначала к WebP, а затем к JPEG/PNG для старых браузеров.
* Устанавливайте явные атрибуты `width` и `height`: установка этих атрибутов позволяет браузеру сразу вычислить соотношение сторон, зарезервировать место и снизить CLS (совокупный сдвиг макета). Размеры изображения могут устанавливаться как атрибутами HTML, так и свойствами CSS.
* Используйте `loading="lazy"` для изображений за пределами экрана: используйте нативную ленивую загрузку браузера для отложенного выполнения сетевых запросов изображений, находящихся за пределами начальной области просмотра.
* Реализуйте отзывчивые изображения с помощью `srcset` и `sizes`: обеспечьте индивидуальное разрешение в зависимости от плотности экрана и ширины области просмотра, чтобы, например, мобильные телефоны не загружали изображения для десктопа.

### Плохо

* Не загружайте лениво изображения, видимые при загрузке страницы: это напрямую ухудшает LCP. Видимые изображения должны использовать `loading="eager"` (дефолтный атрибут).
* Не забывайте про обязательные размеры: отсутствие определения ширины/высоты  у лениво загружаемых изображений приводит к сдвигам макета.
* Не опускайте атрибут `sizes` при использовании атрибута `srcset`: без `sizes` браузер предполагает `100vw` и загружает самое большое изображение из доступных.

### Пример кода

Максимально отзывчивый компонент изображения:

```html
<picture>
  <!-- Современные форматы с согласованием -->
  <source type="image/avif" srcset="hero-400w.avif 400w, hero-800w.avif 800w" sizes="(max-width: 600px) 100vw, 50vw">
  <source type="image/webp" srcset="hero-400w.webp 400w, hero-800w.webp 800w" sizes="(max-width: 600px) 100vw, 50vw">

  <!-- Резерв + размеры + приоритет для изображений на начальной экране -->
  <img
    src="hero-800w.jpg"
    alt="Descriptive text"
    width="800"
    height="600"
    fetchpriority="high"
    loading="eager"
  >
</picture>

<!-- Изображения за пределами экрана -->
<img
    src="footer-icon.png"
    alt="Footer Logo"
    width="100"
    height="100"
    loading="lazy"
>

<!-- Нативная ленивая загрузка iframes за пределами экрана -->
<iframe src="https://example.com/map" width="800" height="600" loading="lazy" title="Example Map"></iframe>
```

## Сервис-воркеры и стратегии кэширования

Кэширование на стороне клиента с помощью сервис-воркеров (Service Workers) позволяет приложениям полностью пропускать сеть, получая ресурсы с диска/памяти.

### Хорошо

* Используйте стратегию `CacheFirst` (сначала кэш) для статичных версионируемых ресурсов: неизменяемые файлы (шрифты, сборки JS/CSS с хэш-строками) должны обслуживаться прямо из кэша для обеспечения мгновенной загрузки.
* Используйте стратегию `StaleWhileRevalidate` (сначала кэш, затем сеть) для динамичных некритичных ресурсов: для вызовов API, когда небольшое устаревание приемлемо, следует сразу возвращать ресурсы из кэша и обновлять его в фоновом режиме.
* Используйте стратегию `NetworkFirst` (сначала сеть) для документов HTML: убедитесь, что пользователь всегда получает последнюю версию страницы или манифест, откатываясь к кэшу только в офлайн режиме.
* Ограничивайте размер и срок жизни кэша: удаляйте ресурсы с истекшим сроком годности, чтобы сервис-воркер не превысил квоту хранилища устройства.

### Плохо

* Не кэшируйте слепо непрозрачные (opaque) ответы: ответы сторонних доменов с отсутствующими заголовками CORS являются "непрозрачными". Их кэширование сильно потребляет квоту и падает бесшумно. Кэшируйте их только с помощью `NetworkFirst` или `StaleWhileRevalidate`.
* Не кэшируйте запросы POST: сервис-воркеры могут нативно кэшировать только GET-запросы. Реализуйте фоновые очереди синхронизации для отправки данных офлайн.
* Не пренебрегайте версионированием: провал обновления хэша/версии ресурса затащит пользователя в бесконечный цикл кэша.

### Пример кода

Кэширование через сервис-воркера с помощью Workbox:

```javascript
import { registerRoute } from 'workbox-routing';
import { CacheFirst, StaleWhileRevalidate, NetworkFirst } from 'workbox-strategies';
import { ExpirationPlugin } from 'workbox-expiration';
import { CacheableResponsePlugin } from 'workbox-cacheable-response';

// 1. Документы HTML: сначала сеть
registerRoute(
  ({ request }) => request.mode === 'navigate',
  new NetworkFirst({ cacheName: 'pages-cache' })
);

// 2. Статичные ресурсы (JS, CSS, шрифты): сначала кэш
registerRoute(
  ({ request }) => ['style', 'script', 'font'].includes(request.destination),
  new CacheFirst({
    cacheName: 'static-resources',
    plugins: [
      new CacheableResponsePlugin({ statuses: [0, 200] }),
      new ExpirationPlugin({ maxEntries: 50, maxAgeSeconds: 30 * 24 * 60 * 60 })
    ]
  })
);

// 3. Ответы API: сначала кэш, затем сеть (возврат ресурса из кэша -> фоновое обновление кэша)
registerRoute(
  ({ url }) => url.pathname.startsWith('/api/v1/content'),
  new StaleWhileRevalidate({
    cacheName: 'api-cache',
    plugins: [
      new CacheableResponsePlugin({ statuses: [0, 200] })
    ]
  })
);
```

## Оптимизация веб-шрифтов

Веб-шрифты являются частым источником блокировки рендеринга. Их оптимизация уменьшает вспышку невидимого текста (Flash of Invisible Text, FOIT) и ускоряет начальный рендеринг.

### Хорошо

* Предварительно загружайте критичные шрифты: используйте `<link rel="preload" as="font" type="font/woff2" crossorigin>` для шрифтов на начальном экране.
* Используйте подмножества шрифтов: удаляйте веса шрифта и варианты глифов, чтобы загружать только символы, которые используются в приложении.

### Плохо

* Не загружайте предварительно все шрифты: это может привести к сетевым конфликтам, мешающим загрузке других критичных ресурсов.

### Пример кода

Определение стиля шрифта:

```css
@font-face {
  font-family: 'Modern Sans';
  src: url('/fonts/modern-sans.woff2') format('woff2');
}
```

Предварительная загрузка критичного шрифта:

```html
<!-- Всегда используем crossorigin для шрифтов, даже если они загружаются из того же источника -->
<link rel="preload" href="/fonts/modern-sans.woff2" as="font" type="font/woff2" crossorigin>
```

## Производительность и метрики видео

Видео также является тяжелым ресурсом. Оптимизация направлена ​​на уменьшение задержки пропускной способности и сохранение стабильности совокупного сдвига макета (CLS).

### Хорошо

* Явно определяйте атрибуты `width` и `height`: установка этих атрибутов резервирует пространство на макете и предотвращает CLS.
* Предоставляйте резервное изображение `poster`: отображайте легковесное изображение-заполнитель во время буферизации видео для улучшения воспринимаемой производительности.
* Используйте `preload="none"` для некритичных видео: откладываем потребление пропускной способности для видео за пределами экрана или видео без автовоспроизведения.
* Используйте современные форматы видео с согласованием: предоставляйте WebM (лучшее сжатие), наряду со стандартным MP4.
* Используйте `loading="lazy"` для видео за пределами экрана: лениво загружаемые видео позволяют откладывать загрузку `poster` и `preload` до подхода видео к области просмотра.

### Плохо

* Не запускайте автоматическое воспроизведение видео без разбора: учитывайте намерение пользователя или используйте потоки прогрессивного улучшения.
* Не запускайте автоматическое воспроизведение больших видеофайлов: полагайтесь на намерение пользователя перед загрузкой больших файлов.

### Пример кода

Доступный и динамичный загрузчик видео:

```html
<video
  controls
  width="1200"
  height="675"
  poster="/images/video-poster.webp"
  preload="none"
>
  <source src="/videos/intro.webm" type="video/webm">
  <source src="/videos/intro.mp4" type="video/mp4">
  <!-- Добавляем субтитры -->
  <track src="/video-caps.vtt" kind="captions" srclang="en" label="English">
</video>
```

## Разделение JavaScript

Тяжелые монолитные сборки блокируют основной поток. Разделение позволяет загружать только те байты, которые нужны для текущего отображения.

### Хорошо

* Используйте динамичные импорты: разделяйте роуты или тяжелые библиотеки UI с помощью стандартных спецификаций `import()`.
* Настраивайте разделение в инструментах сборки: используйте директивы Vite или Webpack для отделения сторонних вендоров от логики приложения.

### Плохо

* Не формируйте одну огромную сборку `app.js`: это увеличивает время парсинга и потребление памяти для начальных отображений.

### Пример кода

Динамичное разделение на основе роута:

```javascript
// Динамичный импорт тяжелого модуля при нажатии кнопки
document.getElementById('heavy-btn').addEventListener('click', async () => {
  const { heavyFunction } = await import('./heavy-module.js');
  heavyFunction();
});
```
