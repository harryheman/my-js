# Руководство по безопасности

Рекомендации по применению превентивных мер безопасности в Интернете.

Обратите внимание: эти рекомендации содержат стандартные меры защиты веб-платформы с акцентом на браузере. Приложениям также требуется защита на стороне сервера, модели авторизации и валидация ввода пользователя.

## Случаи применения этих рекомендаций

Правильная начальная точка зависит от приложения:

- Доработка существующего приложения: всегда начинайте со стадии 1. Применение строгих политик без изучения особенностей приложения просто сломает его. Считайте стадию 2 (только отчеты - report-only) обязательным условием для применения любой методики стадии 3.
- Новое приложение или функция: можно сразу применять принудительные политики стадии 3, подключив отчетность с первого дня.
- Шаблоны SaaS / дефолтные настройки фреймворка: политики стадий 1 и 3 включены по умолчанию, а отчеты стадии 2 включаются для обнаружения регрессий.

Если не уверены, применяйте стадии 1 → 2 → 3 по порядку.

Сосредоточьтесь на главном: в то время как стадии 1 и 2 устанавливают базовый уровень очистки и сбора данных, этап 3 представляет собой работу по обеспечению безопасности с самым высоким уровнем эффективности. В частности, смягчение последствий внедрения кода/XSS с помощью CSP (§3.2) и доверенных типов (Trusted Types) (§3.3) устраняет основную практическую угрозу, в то время как сопутствующие политики и изоляционная защита обеспечивают важную защиту на более глубоком уровне.

## Стадия 1: быстрые решения и очевидные антипаттерны

Перед применением глобальных политик безопасности сфокусируйтесь на гигиене на уровне кода и быстрым исправлениям без риска сломать приложение.

### 1.1. Безопасные контексты

- Передавайте ресурсы по HTTPS для защиты как от пассивных, так и от активных сетевых атак.
- Добавляйте заголовок `Strict-Transport-Security: max-age=31536000; includeSubDomains; preload` для принудительного включения HTTPS, где это возможно.
- При внедрении в продакшн начинайте с короткого `max-age` (например, 300 секунд) и постепенно повышайте его до 1 года. Неправильно настроенный HSTS с длинным `max-age` может привести к рендерингу сайта в недоступном состоянии до истечения времени жизни кэша в каждом браузере, который "видел" сайт в таком состоянии.

### 1.2. Избегайте опасных приемников (sinks) DOM

- Используйте `textContent` или `innerText`, вместо `innerHTML`, при установке текстового контента.
- Используйте `setHTML` (часть Sanitizer API), когда этот метод станет доступным, для безопасной вставки HTML.
- Используйте `innerHTML` или `setHTMLUnsafe` с недоверенным или неочищенным вводом.
- Используйте DOMParser или создавайте элементы программно (`document.createElement`), вместо конкатенации строк HTML.

Опасные приемники: `innerHTML`, `outerHTML`, `document.write`, `eval`, `setTimeout` со строковым аргументом, `script.src`.

```javascript
// Небезопасно
element.innerHTML = `Привет, ${untrustedName}!`;

// Безопасно
element.textContent = `Привет, ${untrustedName}!`;
```

Доверенные типы (Trusted Types) блокируют присвоение строк опасным приемникам во время выполнения кода. Их применение - это этап внедрения CSP с реальным риском поломки приложения.

### 1.3. Безопасные куки

Убедитесь, что новые куки (cookies) имеют безопасные настройки по умолчанию.

- Названия куки должны начинаться с префикса `__Host-`, когда они используются только одним доменом. Это предполагает наличие атрибутов `Secure` и `Path=/` и отсутствие атрибута `Domain`. Это защищает от злоумышленников на одном сайте (same-site) и в сети.
- Названия куки должны начинаться с префикса `__Secure-`, когда `__Host-` не подходит. Это предполагает установку атрибута `Secure` и защищает от злоумышленников в сети.
- Явно устанавливайте `SameSite=Lax` для стандартных основных куки.
- Если ваше приложение встраивается в iframe сторонних контекстов, используйте `SameSite=None; Secure; Partitioned`.
- Не полагайтесь на неразделенный (unpartitioned) `SameSite=None` - они систематически блокируются для предотвращения отслеживания.

```http
Set-Cookie: __Host-session_id=value; SameSite=Lax; HttpOnly; Secure; Path=/
Set-Cookie: third_party_var=value; SameSite=None; Secure; Partitioned
```

### 1.4. Защита от кликджекинга (Frame-Ancestors и X-Frame-Options)

Защита от кликджекинга (clickjacking) проста в применении, имеет очень низкий риск нарушения функционала приложения и обеспечивает немедленную защиту от вредоносной эксплуатации UI.

- Устанавливайте заголовок `X-Frame-Options: SAMEORIGIN` для защиты вашего сайта от встраивания в iframe других сайтов (или используйте `DENY` для полного запрета внедрения).
- Для точечного контроля используйте `frame-ancestors 'self'` (или список доверенных доменов) в заголовке CSP.

```http
X-Frame-Options: SAMEORIGIN
Content-Security-Policy: frame-ancestors 'self' https://trusted-partner.com;
```

### 1.5. Безопасный обмен сообщениями в окне (postMessage)

Если ваше приложение взаимодействует с другими источниками (origins) с помощью `window.postMessage`, вы должны строго валидировать отправителя и получателя.
- Всегда валидируйте `event.origin` входящих сообщений на стороне получателя путем строго сравнения со списком доверенных источников. Не доверяйте подстановочным знакам (`*`) или непроверенной полезной нагрузке (payload).
- Всегда определяйте целевой источник (вместо использования `*`) при вызове `postMessage` для отправки чувствительных данных, чтобы обеспечить их получение только определенным источником.
- Валидируйте и обезвреживайте (sanitize) свойства полезной нагрузки входящих сообщений перед выполнением операций или их записью в приемники DOM. Ручная сериализация JSON необязательна, поскольку `postMessage` клонирует объект автоматически.

```javascript
// Получатель (безопасно - обычная проверка строки)
window.addEventListener('message', (event) => {
  if (event.origin !== 'https://trusted-origin.com') return;
  const data = event.data;
  if (data && data.action === 'update') {
    // Безопасная обработка данных
  }
});

// Отправитель (безопасно)
targetWindow.postMessage({ action: 'update' }, 'https://trusted-origin.com');
```

## Стадия 2: обнаружение и сбор данных (предварительные условия)

Не применяйте строгие политики к существующим приложениям без раздумий. Сначала надо понять ограничения приложения путем сбора данных.

### 2.1. Анализ приложения

Перед настройкой соберите факты:

- Соберите заголовки безопасности из настроек сервера, посредников (middlewares), CDN и мета-тегов: `Content-Security-Policy`, `Strict-Transport-Security`, `X-Frame-Options`, `X-Content-Type-Options`, `Permissions-Policy`, `Cross-Origin-*`, `Access-Control-*`, `Timing-Allow-Origin`, `Reporting-Endpoints`.
- Выявите встроенные скрипты и стили в серверных шаблонах (templates) и в статическом HTML - для них потребуются одноразовые значения (nonces), хэши или рефакторинг.
- Определите скрипты сторонних источников, загружаемые приложением (аналитика, реклама, менеджеры тегов, CDN). От этого зависит, что должен разрешать `script-src`, и является ли `'strict-dynamic'` уместным.
- Определите потоки, зависящие от всплывающих окон: OAuth, платежные шлюзы, SSO. Это ограничивает выбор COOP.
- Определите встраивания из других источников и "встраивателей": iframe, которые загружает приложение, и сайты, на которых приложение внедряется. Это ограничивает COEP/CORP/`frame-ancestor`.
- Перечислите необходимые функции браузера: все функции (камера, геолокация, микрофон, полноэкранный режим), используемые приложением, или встроенные сторонние виджеты для формирования `Permissions-Policy`.
- Определите динамические зависимости: проверьте, имеют ли сторонние скрипты версии или получают ли они автоматические (silent) обновления, чтобы определить, можно ли использовать `SRI`.
- Составьте список межсайтовой интеграции: перечислите все входящие веб-хуки, межсайтовые API или конечные точки перенаправления SSO, чтобы политики изоляции ресурсов `Fetch Metadata` не сломали их.

### 2.2. Развертывание политик только для отчетов

Используйте заголовки "Report-Only" (только для отчетов) для определения потенциальных поломок перед тем, как они произойдут.

- Используйте политики только для отчетов для тестирования политик без их реального применения. Стандартные заголовки только для чтения включают в себя:
  - `Content-Security-Policy-Report-Only` для правил CSP.
  - `Cross-Origin-Opener-Policy-Report-Only` для изоляции COOP.
  - `Cross-Origin-Embedder-Policy-Report-Only` для изоляции COEP.
  - `Document-Policy-Report-Only` для возможностей документа.
- Определите заголовок `Reporting-Endpoints`, чтобы было, куда отправлять нарушения, и ссылайтесь на его имя в `report-to`. Рекомендуется установить конечную точку с именем `default`, которая будет автоматически собирать отчеты об устаревании и сбоях.
- Запускайте такие политики на время, достаточное для охвата реальных моделей трафика (обычно от нескольких дней до нескольких недель), а не только для синтетического тестирования.

```http
Reporting-Endpoints: default="https://reports.example/default", main-endpoint="https://reports.example/main"
Content-Security-Policy-Report-Only: script-src 'nonce-{RANDOM}' 'strict-dynamic' 'report-sample'; object-src 'none'; base-uri 'none'; report-to main-endpoint;
```

Токены `'strict-dynamic'`, `https:` и `'unsafe-inline'` вместе формируют лестницу обратной совместимости: современные браузеры используют `'strict-dynamic'` и игнорируют другие директивы; старые браузеры "откатываются" к `https:`; очень старые браузеры - к `'unsafe-inline'`. Резервные варианты безвредны для любого браузера, поддерживающего более строгие токены.

Конечные точки отчетов получают значительный объем ложноположительных срабатываний о нарушениях, вызванных промежуточным ПО на стороне клиента, агрессивными расширениями браузера, устаревшими браузерами, веб-сканерами или антивирусами. При анализе логов, содержащих только отчеты, сосредоточьтесь на высокочастотных срабатываниях от современных пользовательских агентов и фильтруйте шум, прежде чем принимать решение о развертывании. Конкретно:

- фильтруйте шум: игнорируйте отчеты, отправленные старыми браузерами с известными ошибками, вызывающими ложные срабатывания, отчеты о разметке, которые, как известно, внедряются популярными расширениями браузера или посредниками на стороне клиента (например, идентичные отчеты, наблюдаемые во многих разных приложениях), а также отчеты, которые не содержат достаточно информации для отладки;
- игнорируйте редкие отчеты: если политика развернута, низкий объем нарушений часто указывает на ложное срабатывание, которое можно безопасно игнорировать;
- всегда включайте `'report-sample'` в директивы `script-src`. Это указывает браузеру включать первые 40 символов нарушающего скрипта или встроенного кода в отчет о нарушении, что облегчает отладку.

### 2.3. Очистка данных для отчетов

- Не включайте чувствительные данные (персональные данные, токены аутентификации, идентификаторы сессий, строки запросов с секретами) в логи или отчеты о нарушениях.

### 2.4. Автообнаружение нарушений через браузерные API и инструменты разработчика

- API отчетов: используйте `Reporting-Endpoints` в сочетании с заголовками только для чтения (например, `Content-Security-Policy-Report-Only`, `Document-Policy-Report-Only`) для автоматической отправки нарушений на ваш сервер браузером.
- Инструменты разработчика в браузере: используйте вкладку "Issues" (проблемы) в современных браузерах (например, Chrome DevTools). Браузер автоматически выявляет заблокированные ресурсы, смешанный контент, предупреждения об устаревании сторонних файлов куки и нарушения политики функций без необходимости сканировать базу кода вручную.

## Стадия 3: интерпретация результатов и применение политик

После сбора данных решите, как действовать дальше. Стадия 3 имеет два направления, которые идут параллельно, а не последовательно:

- Основное обеспечение соблюдения (core enforcement) (внедрение на основе данных) - политики с высоким риском поломки приложения, которые зависят от данных стадии 2. Это развертывания, которые основательно готовятся и отслеживаются.
- Сопутствующие политики (companion policies) (параллельное развертывание) - заголовки с низким уровнем риска, которые можно включить одновременно с основной работой или перед ней, почти не требующие проведения мероприятий стадии 2.

### Основное обеспечение соблюдения (внедрение на основе данных)

#### 3.1. Анализ отчетов CSP

При получении отчетов о нарушении CSP, сначала отделите шум (см. §2.2) от реальных проблем приложения. Для нарушений, которые могут быть вызваны особенностями приложения (те, в которых "Sample" или "Blocked URI" выглядят как допустимые скрипты или ресурсы, которые правомерно присутствуют в разметке):

- Поиск кода: найдите в кодовой базе источник, URL-адрес или хеш-код, вызывающий нарушение, и проверьте, присутствует ли он в вашем коде, динамических шаблонах сервера или статических HTML-файлах.
- Аудит консоли: откройте страницу, вызывающую нарушение ("Document URI" в отчете), в том же браузере и проверьте консоль инструментов разработчика на предмет нарушений CSP, тестируя как можно больше функций приложения (некоторые нарушения возникают только в определенных сценариях использования приложения).

После фильтрации и сортировки проанализируйте отчеты по следующим распространенным сценариям:

- Сценарий: множество нарушений встроенных скриптов.
  - Условие: приложение использует фреймворк, полагающийся на большое количество встроенных скриптов.
  - Решение: реализуйте одноразовые значения (nonces) (на сервере) или хэши (статика).
- Сценарий: нарушения для сторонних аналитических скриптов.
  - Условие: скрипты являются обязательными.
  - Решение: используйте `'strict-dynamic'` с одноразовым значением для каждого запроса, чтобы загрузчик аналитики мог зафиксировать свои зависимости. Не добавляйте источник аналитики в список разрешенных URL - белые списки доменов можно обойти с помощью открытых перенаправлений, JSONP и внедрения зависимостей в указанном источнике.
- Сценарий: нарушения доверенных типов на определенных приемниках.
  - Условие: старый код записывает строки через `innerHTML` и т.д.
  - Решение: отрефакторить эти приемники (см. §1.2) или пропускать их через политику доверенных типов (§3.3).

#### 3.2. Переход на принудительное применение CSP

Переходите к принудительному режиму только когда:

1. Нарушения в логах только для отчетов сократились почти до нуля или были учтены.
2. Отчеты остаются подключенными после переключения - сохраняйте `report-to` в принудительном заголовке, чтобы были заметны регрессии.

Ключевые директивы для установки:

- `script-src` с одноразовыми значениями или хэшами - это ключевая директива любого CSP и основной механизм предотвращения XSS.
- `base-uri 'none'` для блокировки перехвата `<base>`. Старые директивы, вроде `object-src 'none'`, могут быть опущены в современных (пост-флеш) веб-окружениях.
- Опционально, но может сломать приложение: `default-src 'self'` иногда используется как запасной вариант для отсутствующих директив получения ресурсов (fetch), но он значительно усложняет развертывание и не имеет большого значения для безопасности, кроме `script-src`. Как правило, безопаснее сначала сосредоточиться на надежном применении `script-src`.
- Опционально: `form-action 'self'` предотвращает отправку форм в источники, контролируемые злоумышленниками.
- Опционально: `upgrade-insecure-requests` заставляет подресурсы HTTP загружаться по HTTPS, хотя современные браузеры делают это по умолчанию для смешанного контента.

Принудительный заголовок CSP с отчетом:

```http
Reporting-Endpoints: main-endpoint="https://reports.example/main"
Content-Security-Policy: script-src 'nonce-{RANDOM}' 'strict-dynamic' 'report-sample'; object-src 'none'; base-uri 'none'; report-to main-endpoint;
```

HTML для основанного на одноразовом значении CSP:

```html
<script nonce="{RANDOM}" src="https://example.com/script.js"></script>
```

Для статического/кэшированного HTML (SPA), когда одноразовые значения для каждого запроса невозможны, используйте основанный на хэше CSP: хэшируйте каждый встроенный скрипт и указывайте хэши в `script-src`.

Избегайте белых списков URL, вроде `script-src https://cdn.example.com` - их легко обойти с помощью открытых перенаправлений, конечных точек JSONP и внедрения зависимостей в разрешенном источнике.

#### 3.3. Принудительное применение доверенных типов

Доверенные типы реализуют рекомендации §1.2 на уровне источника во время выполнения: после включения браузер блокирует присвоение строк опасным приемникам, если они не проходят через именованную политику.

- Стратегия постепенного применения: хотя полное соблюдение требований сопряжено с реальным риском поломки приложения, не требуется делать все сразу. Весьма жизнеспособный подход - определить и внедрить политику для небольшой части приложения и постепенно расширять ее использование по мере замены приемников. Это упрощает глобальное применение без риска поломки приложения.
- Предварительные условия: доверенные типы требуют взаимодействия с фреймворком. Если фреймворк приложения (или любой сторонний виджет, который записывает значения в приемники DOM) не предоставляет значений `TrustedHTML` / `TrustedScript`, политика не может применяться без поломки этого кода. Проверьте поддержку фреймворка перед стадией 2.
- Предварительные условия: рефакторинг приемника на уровне кода, проведенный на стадии 1, является обязательным условием для полного применения доверенных типов (стандартное применение `script-src` CSP, напротив, не контролирует приемники DOM и может применяться без их рефакторинга).
- Сначала выполните развертывание с заголовком `Content-Security-Policy-Report-Only: require-trusted-types-for 'script'`, чтобы найти все нарушающие правила приемники.
- Определите одну именованную политику, выполняющую обезвреживание и пропускайте запись во все приемники через нее.
- Переходите к полному глобальному применению политики `Content-Security-Policy: require-trusted-types-for 'script'`, как только политика будет успешно интегрирована, а нарушения в логах только для отчетов снизятся до нуля.

```javascript
if (window.trustedTypes && trustedTypes.createPolicy) {
  const policy = trustedTypes.createPolicy('escapePolicy', {
    createHTML: str => str.replace(/</g, '&lt;').replace(/>/g, '&gt;')
  });
  el.innerHTML = policy.createHTML(untrustedString);
}
```

#### 3.4. Cross-Origin Opener Policy (COOP)

Самый низкий риск поломки приложения из трех. Смело применяйте эту политику, если приложение не является провайдером OAuth, обработчиком платежей или иным образом не предполагает доступа к нему "открывающего" (opener).

- Используйте `Cross-Origin-Opener-Policy: same-origin-allow-popups` - не позволяет злоумышленнику организовать атаки через XS-утечки, в то же время разрешая OAuth и платежи, которые инициирует само приложение.
- Не переходите к `same-origin`, если явно не убедились, что ни одна интеграция не опирается на доступ к `window.opener` из разных источников.

#### 3.5. Cross-Origin Resource Policy (CORP)

Явно задайте CORP для каждого ответа в зависимости от того, следует ли его встраивать в другие контексты. Два основных преимущества: он защищает ресурсы от вредоносного чтения из разных источников и обеспечивает совместимость, когда страницы запрашивают более сильную изоляцию на стороне клиента.

- Используйте `Cross-Origin-Resource-Policy: same-origin` для внутренних ресурсов приложения (данные аутентификации, сессия пользователя в формате JSON, ограниченные внутренние скрипты).
- Используйте `same-site` для конечных точек, используемых между поддоменами одного eTLD+1 (Effective Top-Level Domain плюс один уровень - это та часть веб-адреса, которая определяет реального владельца сайта (регистрационный домен). Она состоит из доменного имени второго (или третьего) уровня и суффикса (eTLD)).
- Используйте `cross-origin` исключительно для ресурсов, созданных для общего внедрения или широко кэшированной доставки (например, статические общие ресурсы или общедоступные CDN).

#### 3.7. Получение метаданных (изоляция ресурсов)

Серверная политика, которая использует заголовки запросов `Sec-Fetch-*` для отклонения подозрительных межсайтовых запросов. Перед применением требуется сопоставление межсайтовой интеграции из §2.1.

- Реализуйте серверную политику изоляции ресурсов, которая проверяет заголовки `Sec-Fetch-*` и отклоняет запросы `cross-site` для конечных точек, не связанных с навигацией.
- Отклоняйте запрещенные запросы перед проверкой аутентификации или авторизации, чтобы ответ не содержал информацию о времени существования ресурса или сессии.
- Включайте `Vary: Sec-Fetch-Dest, Sec-Fetch-Mode, Sec-Fetch-Site` для предотвращения предоставления кэшированных ответов злоумышленникам от промежуточных кэшей (CDN).
- Предупреждение: `same-site` доверяет всем поддоменам вашего eTLD+1. Если в каком-либо поддомене размещен пользовательский контент, устаревшее приложение или иной ненадежный код, удалите `same-site` из белого списка и принимайте только `same-origin` и `none`.
- Предупреждение: неправильная настройка этих проверок приведет к блокировке законных запросов API из других источников, обработчиков SSO или веб-хуков. Обязательно протестируйте ограничения `Sec-Fetch-*` перед продакшн-развертыванием.

```javascript
app.use((req, res, next) => {
  const site = req.get('Sec-Fetch-Site');
  const mode = req.get('Sec-Fetch-Mode');
  const dest = req.get('Sec-Fetch-Dest');

  if (!site) return next(); // Резерв для старых браузеров

  if (['same-origin', 'same-site', 'none'].includes(site)) return next();

  // Разрешаем стандартные GET-запросы навигации (клики по ссылкам)
  if (site === 'cross-site' && mode === 'navigate' && req.method === 'GET' && !['object', 'embed'].includes(dest)) {
    return next();
  }

  res.status(403).send('Forbidden');
});
```

### Сопутствующие политики (параллельное развертывание)

Эти политики несут значительно меньший риск поломки приложения. Их можно применять одновременно или до применения CSP и изоляции.

#### HTTP Strict Transport Security (HSTS)

- Используйте `Strict-Transport-Security: max-age=31536000; includeSubDomains; preload` для принудительного включения HTTPS.
- Подсказка: в продакшне начинайте с короткого `max-age` (например, 300 секунд) и постепенно увеличивайте значение до 1 года. Неправильно настроенный HSTS с длинным `max-age` может привести к рендерингу сайта в недоступном состоянии до устаревания кэша в каждом браузере, который "видел" сайт в таком состоянии.

#### X-Content-Type-Options

- Устанавливайте `X-Content-Type-Options: nosniff` для блокировки анализа контента для определения его MIME-типа.
- Убедитесь, что сервер возвращает корректные заголовки `Content-Type` для всех ресурсов (`application/javascript` для скриптов, `application/json` для API, `text/html` для документов и т.д.), чтобы браузер мог строго применять ограничение `nosniff`.

#### Referrer Policy

- Используйте `Referrer-Policy: strict-origin-when-cross-origin` в качестве дефолтной политики безопасности.

#### Permissions Policy

- Отключайте неиспользуемые возможности браузера (камера, геолокация, микрофон) для страниц и iframe с помощью синтаксиса структурированных полей (Structured Fields).
- При делегировании возможностей iframe, используйте атрибут `allow` в HTML в дополнение к заголовку.
- Предупреждение: непреднамеренная блокировка делегированных возможностей приведет к "тихим" провалам в сторонних виджетах (таким как встраиваемые видеоплееры или платежные шлюзы). Проверяйте сторонние зависимости перед блокировкой.

```http
Permissions-Policy: camera=(), geolocation=(), microphone=()
```

```html
<iframe src="https://trusted-video.com/player" allow="fullscreen; camera"></iframe>
```

#### Subresource Integrity (SRI)

- Используйте атрибут `integrity` с криптографическим хэшем (предпочтительно `sha256` или `sha512`) при загрузке сторонних скриптов, в сочетании с `crossorigin="anonymous"`.
- Убедитесь в том, что сервер/CDN отправляет правильный заголовок `Access-Control-Allow-Origin`, чтобы браузер мог вычислить хэш.
- Не используйте SRI для динамических или неверсионируемых ресурсов - автоматические обновления приведут к провалу выполнения скрипта. SRI предназначен только для неизменяемых, версионируемых ресурсов.

```html
<script src="https://cdn.example.com/lib.js" integrity="sha256-H8df...39v" crossorigin="anonymous"></script>
```

#### Cross-Origin Resource Sharing (CORS)

CORS - это разрешение, а не защита - он сообщает браузеру, какие источники имеют доступ к API. Риск заключается в слишком либеральной настройке.

- Валидируйте заголовок `Origin` на сервере и устанавливайте `Access-Control-Allow-Origin` динамически для определенных источников (вместо использования подстановочного знака `*`).
- Не используйте `*` для `Access-Control-Allow-Origin`, если `Access-Control-Allow-Credentials: true` является обязательным - браузер отклонит ответ.
- Обрабатывайте запросы `OPTIONS`, возвращая правильные заголовки перед обработкой данных.

```http
Access-Control-Allow-Origin: https://trusted-app.com
Access-Control-Allow-Credentials: true
```

#### Clear-Site-Data (выход из системы)

- Используйте `Clear-Site-Data` для конечных точек выхода из системы (logout) для полного завершения сессии:

```http
Clear-Site-Data: "cookies", "storage", "cache"
```
