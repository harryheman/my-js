## 1. Основы

По-возможности, полагайтесь на движок компоновки браузера ради лучшей производительности. Используйте внутренние (intrinsic) размеры, логические свойства и `aspect-ratio`, прежде чем прибегать к жесткому кодированию размеров или сложным медиа-запросам.

### 1.1. Какой режим компоновки использовать?

Обходите дерево решений сверху вниз и останавливайтесь на первом совпадении. Обратите внимание, что макеты могут быть вложенными друг в друга, и каждое решение должно основываться на специфике использования конкретного контейнера.

1. Это простая строка или колонка элементом? Используйте `flex` - одномерная компоновка с приоритетом контента, контент располагается вдоль одной оси.
2. Должен ли вложенный элемент совпадать с дорожками родительской сетки? Используйте `subgrid` - двумерная компоновка с приоритетом связей, наследует родительские треки, чтобы внуки могли выравниваться с сиблингами.
3. Это сложная страница или компонентная структура со строками и колонками? Используйте `grid` - двумерная компоновка с приоритетом макета, мы определяем скелет, а контент заполняет его.
4. Контент является длинным текстом, который должен быть разделен на сбалансированные колонки? Используйте `columns` - одномерная компоновка в стиле газет.
5. Элементы имеют разную высоту и должны быть плотно упакованы? Пока используйте `grid` с `grid-auto-flow: dense`; скоро можно будет использовать нативную кладку (masonry) (так называемую "дорожки сетки" (grid lanes)) (см. раздел 8).
6. Должен ли элемент плавать над страницей и оставаться пространственно привязанным к триггеру, даже за пределами границ DOM или стековых контекстов? Используйте якорное позиционирование (anchor positioning) - `anchor-name` на триггере, `position-anchor` на оверлее (см. раздел 5).

### 1.2. Рабочие принципы

Хорошо:

- используйте логические свойства (`inline-size`, `block-size`, `margin-inline`, `padding-block`, `inset-inline-start`) для направлений и отступов;
- применяйте подходящую ментальную модель: `flex`, когда элементы определяют поток, `grid`, когда мы сначала определяем скелет;
- используйте сокращения `place-*` (`place-content`, `place-items`, `place-self`) для выравнивания элементов по обеим осям в одном объявлении;
- используйте внутренние размеры (`min-content`, `max-content`, `fit-content()`) и гибкие дорожки (`fr`, `minmax()`), вместо фиксированных `width`/`height` - меньше медиа-запросов, более гибкие макеты;
- используйте `aspect-ratio` для резервирования места для медиа и предотвращения сдвигов макета перед загрузкой ресурсов.

```css
.sidebar       { inline-size: max-content; }    /* Растягивание до самого длинного неделимого токена. */
.main-content  { inline-size: fit-content; }    /* Растягивание до доступного пространства, не больше. */
.media         { aspect-ratio: 16 / 9; inline-size: 100%; block-size: auto; }
body.centered  { display: grid; place-content: center; min-block-size: 100dvb; }
```

## 2. Flexbox

Однонаправленная компоновка - элементы располагаются вдоль одной основной оси с выравниванием по другой оси. Используется для навигационных панелей, панелей инструментов, строк элементов и любого однострочного или одноколоночного расположения элементов.

Хорошо:

- Определяйте контекст с помощью `display: flex` и основную ось с помощью `flex-direction` (`row` по умолчанию).
- Используйте `flex-wrap: wrap`, когда существует возможность переполнения - в случае `nowrap` без `overflow: auto/hidden` элементы будут "выпадать" на узких экранах.
- Используйте сокращение `flex: <grow> <shrink> <basis>` (например, `flex: 1 1 250px`), вместо индивидуальной установки `flex-grow`/`flex-shrink`/`flex-basis`.
- Используйте `gap` (или `row-gap`/`column-gap`) для отступов между элементами, вместо `margin`.
- Добавляйте префикс `safe` к позиционным аргументам (например, `align-items: safe center`), чтобы фокусируемый контент не обрезался, когда контейнер уже, чем его контент.
- Помещайте одинокий элемент в самый конец основной оси с помощью `margin-inline-start: auto` (или `margin-block-start: auto`) - это стандартный резервный трюк.
- Перезаписывайте выравнивание по оси для элемента с помощью `align-self`.
- Используйте `align-items` для центрирования всех элементов по поперечной оси; используйте `margin: auto` на элементе для его независимого центрирования по обеим осям; используйте `align-content`, только когда контейнер переносится и между строками появляется дополнительное пространство.
- Устанавливайте `min-inline-size: 0` (или `min-width: 0`) на элементах, которые содержат длинный неделимый контент (URL, код, длинные строки) - флекс-элементы по умолчанию не сужаются меньше размера своего контента, что приводит к переполнению.

Плохо:

- Не используйте `justify-self` на флекс-элементах - это работает только на сеточных, блочных и абсолютно позиционируемых макетах. Используйте `margin: auto`.
- Не используйте `order` или `flex-direction: *-reverse` для переупорядочения интерактивного контента. Они меняют только визуальный порядок; порядок DOM остается прежним, поэтому порядок фокусировки на элементах с помощью `Tab` не будет соответствовать тому, что видит пользователь.
- Не путайте `space-around` (половина отступа на концах) с `space-evenly` (равные отступы до, между и после).
- Не забывайте о переворачивании осей: в случае `flex-direction: column`, `justify-content` выравнивает элементы по блочной оси, а `align-items` - по строчной (по умолчанию - наоборот).
- Не устанавливайте размер контейнера и его дочерних элементов так, чтобы они заполняли друг друга - это распространенный источник переполнения и неожиданных результатов.
- Не устанавливайте и `flex-basis`, и `width`/`inline-size` на одном элементе - `flex-basis` имеет приоритет во флекс-контексте и `width` будет игнорироваться. Используйте `flex-basis` (или сокращение `flex`) как единственный источник истины для размеров флекс-элементов.

```css
.card-grid        { display: flex; flex-flow: row wrap; gap: 1rem; }
.card-item        { flex: 1 1 250px; }                  /* grow, shrink, basis */
.card-item-action { margin-inline-start: auto; }        /* Помещаем в конец основной оси. */
.toolbar          { display: flex; align-items: safe center; }
```

## 3. Grid и subgrid

Subgrid получил полную поддержку основных браузеров 15.09.2023.

Двунаправленная компоновка - определяйте строки и колонки явно или позвольте сделать это движку. Subgrid позволяет вложенной сетке наследовать дорожки предка, чтобы внуки выравнивались по сиблингам.

Выбор возможностей сетки:

- Знаем ли мы точное количество колонок, которые нам нужны?
  - Да - используйте явные дорожки (`grid-template-columns: 200px 1fr`, `repeat(3, 1fr)` и т.д.).
    - Колонки должны иметь разные размеры (боковая панель + основной контент, шапка во всю ширину области просмотра)? → используйте `grid-template-areas` для именованных областей.
    - Все колонки одинаковые или позиционируются по номеру строки? → используйте `repeat(N, ...)` или именованные строки.
  - Нет (отзывчивый макет, неизвестное число элементов) - используйте `repeat(auto-fit, minmax(min, 1fr))`.
    - Должны ли элементы на последней строке заполнять оставшееся пространство? → `auto-fit`.
  -  Должны ли пустые дорожки последней строки сохранять свой минимальный размер (с сохранением "призрачных" слотов столбцов)? → `auto-fill`.
- Требуется ли поместить элемент в определенную локацию?
  - Да - используйте `grid-column: <start> / <end>` или `grid-area: <name>`.
  - Нет (просто растягивается на несколько дорожек, положение в потоке не имеет значения) - используйте `grid-column: span <n>`.
- Должны ли элементы наследовать размеры сетки предка (неравномерное выравнивание между сиблингами)?
  - Да - используйте `subgrid` на соответствующей оси
    - Количество потомков в ячейках варьируется? → `subgrid` для одной оси, `grid-auto-rows`/`grid-auto-columns` - для другой.
    - Количество потомков фиксированное? → `subgrid` для обеих осей.
  - Нет - обычный `grid`, `subgrid` не нужен.

Хорошо:

- Устанавливайте контекст с помощью `display: grid`.
- Используйте `grid-template-areas` для сложных макетов уровня страницы - названия областей являются самодокументируемыми, и объявление может содержать выравнивание по строкам и колонкам для удобства чтения.
- Используйте `repeat(auto-fit, minmax(200px, 1fr))` для адаптивных сеток карточек, которые растягивают заполненные дорожки, чтобы заполнить строку, или `auto-fill`, чтобы сохранить минимальный размер пустых повторяющихся дорожек.
- Используйте `fr` для пропорционального распределения дорожек и `minmax(min, max)` для гибких, но ограниченных дорожек.
- Позиционируйте элементы с помощью `grid-column: span <n>` для растягивания элемента на несколько дорожек, `grid-column: <start> / <end>` для помещения элемента в определенное место или `grid-area: <name>` для именованных областей.
- Используйте `subgrid` (`grid-template-columns: subgrid` или `grid-template-rows: subgrid`) для решения проблемы "рваных краев" в списках карточек - внутренние элементы, такие как заголовки, метаданные и CTA (Call to Action - призыв к действию), выравниваются между сиблингами.
-  Для `subgrid` предусмотрите дополнительное явное объявление `grid-template-rows`/`-columns` в качестве резервного варианта для старых браузеров.

Плохо:

- Не ждите, что размер дорожек `auto-fit`/`auto-fill` вычисляется на основе контента элемента - он берется из аргумента размера `repeat()`.
- Не используйте `grid-auto-flow: dense` для интерактивного контента. Это свойство эффективно упаковывает элементы, но переупорядочивает их визуально, ломая поток переключение между элементами с помощью `Tab`, который определяется порядком элементов в DOM.
- Не применяйте `subgrid` к обеим осям, когда количество потомков варьируется. Дополнительные элементы попадают в последнюю дорожку; используйте `grid-auto-rows`/`grid-auto-columns` для неявной оси.
- Не путайте `justify-items`/`align-items` (выравнивают контент элемента внутри его дорожки) с `justify-content`/`align-content` (выравнивают дорожки сетки внутри контейнера).
- Не используйте `repeat(auto-fit/auto-fill, ...)` без определения `inline-size` для контейнера - внутри `display: inline-grid` или безразмерного флекс-элемента, контейнер не будет иметь ширину для разделения, поэтому количество дорожек будет непредсказуемым.

### 3.1. Пример кода: grid и subgrid

Структура страницы: `<main class="page-layout">` содержит потомков `<header>`, `<aside>`, `<section class="card-grid">` с `<div class="card">` и `<footer>`.

```css
/* Структурируем grid-template-areas по строкам и колонкам для улучшения читаемости. */
.page-layout {
  display: grid;
  grid-template-columns: repeat(3, 1fr);
  grid-template-areas:
    "header  header  header"
    "sidebar main    main"
    "footer  footer  footer";
  gap: 1.5rem;
}

header  { grid-area: header; }
aside   { grid-area: sidebar; }
footer  { grid-area: footer; }

.card-grid {
  grid-area: main;
  display: grid;
  grid-template-columns: repeat(auto-fill, minmax(240px, 1fr));
  grid-template-rows: auto 1fr; /* title block, body block */
  gap: 1rem;
}

.card {
  grid-row: span 2;
  display: grid;
  /* Резерв: игнорируется в случае поддержки subgrid. */
  grid-template-rows: auto 1fr;
  grid-template-rows: subgrid;
}
```

## 4. Запросы контейнера

Запросы контейнера (container queries) получили полную поддержку основных браузеров 14.02.2023.

Позволяют запрашивать размер (или вычисленный стиль) родительского контейнера, вместо области просмотра. Ментальная модель: запросы контейнера = контекст компонента; медиа-запросы = глобальный макет страницы и настройки пользователя (`prefers-color-scheme`, `prefers-reduced-motion`).

Хорошо:

- Устанавливайте контекст сдерживания (containment context) с помощью `container-type: inline-size` (запросы только ширины) или `container-type: size` (обе оси) на контейнере-обертке.
- Именуйте контейнеры с помощью `container-name` (или сокращения `container`: `container: inline-size card`), когда вложенные контексты могут конфликтовать между собой.
- Включайте единицы измерения запросов контейнера в расчеты типа гибкой типографики и расстояний: `cqi`/`cqb` (логический инлайн/блок), `cqw`/`cqh` (физический), `cqmin`/`cqmax`.
- Явно определяйте `block-size` контейнера при любом использовании `container-type: size` - без этого потомки будут коллапсировать, поскольку сдерживание размера заставляет контейнер игнорировать его контент.

Плохо:

- Не используйте `block-size` как значение `container-type` - оно невалидно. Используйте `size` для обеих осей.
- Не ждите, что внутренний размер потомков повлияет на контейнер после объявления `container-type`. При сдерживании размеры контейнера вычисляются так, будто у него нет потомков, когда сдерживание активно.
- Не полагайтесь на единицы измерения запросов контейнера в потомках неквалифицированного предка; они откатятся к `svw`/`svh`.

### 4.1. Пример кода: гибкая типографика с помощью единиц измерения запросов контейнера

```css
.card-wrapper {
  container: inline-size / card; /* сокращение для container-type + container-name */
}

@container card (inline-size > 400px) {
  .content {
    display: flex;
    gap: 2rem;
  }
}

.title {
  /* Размеры шрифта определяются шириной контейнера, а не области просмотра. */
  font-size: clamp(1rem, 4cqi, 2rem);
}
```

## 5. Нативные оверлеи, якорное позиционирование и стековые контексты

Элемент `<dialog>` получил полную поддержку основных браузеров 14.03.2022. Popover API - 27.01.2025. Якорное позиционирование (anchor positioning) полной поддержки пока не получило.

Случаи использования каждого "оверлейного" примитива:

- Используйте `popover` для скоротечного немодального UI (выпадающие меню, уведомления, всплывающие подсказки) - "живут" на верхнем слое (top layer), управление `z-index` не требуется.
- Используйте `<dialog>` с `.showModal()` для модальных взаимодействий, требующих захват фокуса и инертный фон.
- Не используйте `popover` и `.showModal()` на одном элементе - это взаимоисключающие состояния времени выполнения.

Якорное позиционирование (пространственное расположение оверлеев):

- Используйте `position-area` (или `anchor()`) и `anchor-size()` для позиционирования и масштабирования оверлея относительно его якоря.
- Используйте `position-try-fallbacks: flip-block` (или `flip-inline`) для разрешения браузеру менять положение оверлея при его выходе за пределы области просмотра.
- Не смешивайте физические и логические ключевые слова в одном значении `position-area` - используйте одну координатную систему.
- Определяйте поддержку с помощью `@supports (anchor-name: --x)` и предоставляйте резерв на основе абсолютного позиционирования.

## 6. Отслеживание переполнения и стабильность макета

Свойство `scrollbar-gutter` получило полную поддержку основных браузеров 11.24.2024. Свойство `line-clamp` полной поддержки пока не получило.

Указанные свойства позволяют управлять сдвигами макета, скроллбарами и делать обрезку контента более предсказуемой.

Хорошо:

- Используйте `overflow: auto`, чтобы скроллбары появлялись только при реальном переполнении контента.
- Используйте `overflow: clip` для обрезки контента без создания прокручиваемого контейнера; управляйте обрезаемой областью с помощью `overflow-clip-margin`.
- Используйте `scrollbar-gutter: stable` для резервирования пространства для скроллбаров во избежание сдвигов макета при увеличении контента.
- Используйте `overscroll-behavior: contain` (или `none`) на прокручиваемых контейнерах для остановки цепочек прокрутки от всплытия до предка или документа.
- Используйте триаду `-webkit-line-clamp` + `display: -webkit-box` + `-webkit-box-orient: vertical` для многострочной обрезки текста - несмотря на префикс, этот паттерн поддерживается всеми основными браузерами. Определяйте сокращение `line-clamp` без префикса; браузеры, которые пока его не поддерживают, просто его проигнорируют.

Плохо:

- Не используйте `overflow: scroll`, когда можно использовать `auto` - `scroll` приводит к отображению скроллбаров, даже когда прокручивать нечего.
- Не используйте `overflow: hidden`, когда вы хотите просто обрезать контент - `hidden` создает прокручиваемый контейнер, который можно прокручивать программно.

```css
.scrollable-list {
  max-block-size: 400px;
  overflow-y: auto;
  scrollbar-gutter: stable;       /* Резерв пространства для скроллбара. */
  overscroll-behavior: contain;   /* Отключаем цепочку прокрутки на странице. */
}

.snippet {
  display: -webkit-box;
  -webkit-line-clamp: 3;
  -webkit-box-orient: vertical;
  line-clamp: 3;                  /* Игнорируется при отсутствии поддержки. */
  overflow: clip;
}
```

## 7. Механики области просмотра и распределение дорожек

Малые, большие и динамические единицы измерения области просмотра получили полную поддержку основных браузеров 05.12.2022.

- Используйте `dvh`/`dvw` для мобильных макетов контейнеров, которые должны учитываться сдвиги браузерного UI (раскрытие/скрытие панели URL).
- Не используйте `100vw` для макета полной ширины - ширина скроллбара будет игнорироваться, что приведет к горизонтальному переполнению. Используйте `100%`, `100dvw` или `100svw`.

## 8. Сетчатые дорожки (также известные как кладка)

Кладка (masonry) пока не поддерживается ни одним из основных браузеров.

Спецификация находится в разработке. Текущее рабочее название - "дорожки сетки" (grid lanes) (например, `display: grid-lanes`). Firefox поддерживает `grid-template-rows: masonry` за флагом.

Хорошо:

- Пока используйте `grid` с `grid-auto-flow: dense` для плотной упаковки элементов, принимая во внимание, что порядок элементов в DOM может не совпадать с их визуальным порядком.
- Используйте несколько колонок (`columns: 3; column-gap: 1rem`) для содержащего много контента и похожего на кладку потока, когда элементы являются фрагментами документа, а не карточками.
- Используйте `grid-template-rows: masonry` только в качестве прогрессивного улучшения - определяйте поддержку с помощью `@supports`.

Плохо:

- Не используйте `grid-template-rows: masonry` без резерва до полной поддержки этой возможности всеми основными браузерами.

```css
.gallery       { columns: 3 200px; column-gap: 1rem; }
.gallery > *   { break-inside: avoid; margin-block-end: 1rem; }

@supports (grid-template-rows: masonry) {
  .gallery {
    display: grid;
    grid-template-columns: repeat(auto-fill, minmax(200px, 1fr));
    grid-template-rows: masonry;
    gap: 1rem;
    columns: unset;
  }
}
```
