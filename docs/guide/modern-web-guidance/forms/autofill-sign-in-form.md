# Форма авторизации, соответствующая лучшим практикам

Используйте кроссплатформенные возможности браузеров для разработки безопасных, доступных и легких в использовании форм авторизации.

Если пользователям требуется вход на ваш сайт, качественный дизайн формы авторизации имеет решающее значение. Это особенно актуально для людей с плохим интернет-соединением, использующих мобильные телефоны, спешащих или находящихся в состоянии стресса. Плохо разработанные формы авторизации приводят к высокому показателю отказов. Каждый отказ может означать потерю клиента и недовольного пользователя, а не просто упущенную возможность авторизации.

## Реализация

Ниже перечислены наиболее важные рекомендации по созданию хороших форм авторизации.

### Используйте осмысленный валидный HTML

Получайте максимальную выгоду от встроенных элементов и атрибутов форм:

-   `<form>`, `<input>`, `<label>` и `<button>`
-   `type`, `autocomplete` и `inputmode`

Это включает встроенный браузерный функционал, улучшает доступность и добавляет осмысленности в разметку.

### Используйте элемент `<label>` для добавления подписей к полям формы

Для добавления подписей к `<input>`, `<select>` или `<textarea>` используйте `<label>`. Связывайте подпись с инпутом путем добавления атрибуту `for` подписи и атрибуту `id` инпута одинакового значения.

### Получайте максимальную выгоду от атрибутов HTML

Облегчите пользователям ввод данных путем использования подходящего атрибута `<type>` элемента `<input>` для предоставления правильной клавиатуры на мобильных телефонах и включения базовой встроенной браузерной валидации.

Всегда используйте `type="email"` для адресов электронной почты и `type="tel"` для номеров телефона:

```html
<!-- type="email"/"tel" дает пользователям правильную мобильную клавиатуру и включает встроенную валидацию -->
<input type="email" id="email" name="email" autocomplete="email" required>
<input type="tel" id="phone" name="phone" autocomplete="tel">
```

Каждый элемент `<input>`, `<select>` и `<textarea>` должен содержать подходящий атрибут `autocomplete` для улучшения доступности и помощи пользователям в повторном вводе данных.

### Сделайте кнопки полезными

Используйте `<button>` для кнопок. Вы также можете использовать `<input type="submit">`, но не используйте `div` или другой элемент в качестве кнопки. Элементы `<button>` предоставляют доступное поведение, встроенный функционал отправки форм и легко стилизуются.

Предоставляйте каждой кнопке для отправки формы понятное название. Используйте ясную легко распознаваемую подпись. Например, используйте "Войти", вместо "Продолжить" или "Отправить".

### По возможности используйте одно поле для ввода имени

Позвольте пользователям вводить свое имя с помощью одного поля, если у вас нет веской причины хранить имена, фамилии, почетные звания или другие части имени отдельно. Использование одного поля ввода имени упрощает заполнение форм, позволяет копировать и вставлять данные, а также упрощает автозаполнение.

Разрешите использование международных имен. Для валидации избегайте использования регулярных выражений, которые соответствуют только латинским символам. Использование только латинских символов исключает пользователей, чьи имена или адреса содержат символы, не входящие в латинский алфавит. Вместо этого разрешите сопоставление с символами юникода - и убедитесь, что ваш бэкэнд поддерживает юникод как на входе, так и на выходе. Юникод в регулярных выражениях хорошо поддерживается современными браузерами.

### Показывайте прогресс авторизации

Для каждого шага авторизации используйте заголовки страниц и описательные названия кнопок, которые сообщают, что нужно сделать и каким будет следующий шаг.

Используйте атрибут `enterkeyhint` в полях ввода формы для установки метки клавиши Enter на мобильной клавиатуре. Например, используйте `enterkeyhint="назад"` и `enterkeyhint="вперед"` в многостраничной форме, `enterkeyhint="готово"` - для финального инпута формы и `enterkeyhint="поиск"` - для поискового инпута.

### Помогайте пользователям в повторном вводе данных

Убедитесь в добавлении подходящих значений `autocomplete` в формы авторизации.

Это позволяет браузерам помогать пользователям, безопасно сохраняя данные для входа и корректно вводя их в форму. Без автозаполнения пользователи с большей вероятностью будут хранить бумажные записи с данными для входа или небезопасно хранить их на своем устройстве.

### Валидируйте осторожно

Валидируйте данные в реальном времени и перед отправкой формы. Используйте `type="email"` для инпутов адресов электронной почты - браузер будет выполнять валидацию автоматически. Добавляйте атрибут `required` к обязательным полям для предотвращения отправки пустых полей.

### Используйте элемент `<form>` для входа в систему

Всегда используйте элемент `<form>` для ввода пользовательских данных.

Не оборачивайте поля ввода в `<div>` и не обрабатывайте отправку данных исключительно с помощью JavaScript. Как правило, лучше использовать элемент `<form>`. Это делает ваш сайт доступным для программ чтения с экрана и других вспомогательных устройств, обеспечивает доступ к ряду встроенных функций браузера, упрощает создание базового функционала авторизации для старых браузеров, который продолжает работать даже при отключении JavaScript.

### Не дублируйте инпуты

Некоторые сайты заставляют пользователей вводить электронные адреса или пароли дважды. Это может уменьшить количество ошибок у некоторых пользователей, но создает дополнительную работу для всех и увеличивает процент отказов. Повторный запрос также не имеет смысла, если браузеры автоматически заполняют адреса электронной почты или предлагают надежные пароли. Лучше предоставить пользователям возможность подтвердить свой адрес электронной почты (вам все равно придется это сделать) и упростить им сброс пароля при необходимости.

### Храните пароли в секрете, но разрешите пользователям просматривать их при желании

Инпуты для паролей должны содержать атрибут `type="password"` для скрытия текста пароля и помощи браузерам в понимании того, что инпут предназначен для ввода пароля (браузеры используют различные методы для определения ролей инпутов и принятия решения о том, предлагать ли сохранение паролей).

Следует добавить переключатель "Показать пароль", чтобы пользователи могли проверить введенный текст, и не забудьте добавить ссылку "Забыли пароль?".

### Предоставляйте пользователям правильную мобильную клавиатуру

Используйте `<input type="email">` для предоставления мобильным пользователям правильной клавиатуры и включения базовой встроенной браузерной валидации адреса электронной почты... JavaScript не требуется!

Если нужен номер телефона, `<input type="tel">` включает телефонную клавиатуру на мобильных телефонах. Также можно использовать атрибут `inputmode` при необходимости: например, `inputmode="numeric"` идеален для PIN-кода.

### Предотвратите блокировку кнопки "Вход" мобильной клавиатурой

Если не быть осторожным, мобильная клавиатура может закрыть форму или, что еще хуже, частично перекрыть кнопку "Войти". Пользователи могут сдаться, даже не поняв, что произошло.

По возможности избегайте этого, отображая только поля для ввода адреса электронной почты (или номера телефона) и пароля, а также кнопку "Войти" в верхней части страницы входа. Остальное содержимое размещайте ниже.

### Помогайте пользователям в повторном вводе данных

Вы можете помочь браузерам правильно сохранять данные и автоматически заполнять поля ввода, чтобы пользователям не приходилось запоминать введенные адрес электронной почты и пароль. Это особенно важно на мобильных устройствах и крайне необходимо для полей электронного адреса, которые часто не заполняются. Здесь есть две составляющие:

1.  Атрибуты `autocomplete`, `name`, `id` и `type` помогают браузерам понять роли инпутов для сохранения данных, которые позже могут использоваться для автозаполнения. Для сохранения данных для автозаполнения современные браузеры также требуют, чтобы поля ввода имели стабильное значение `name` или `id` (а не генерировались случайным образом при каждой загрузке страницы или развертывании сайта) и находились в элементе `<form>` с кнопкой `submit`.
2.  Атрибут `autocomplete` помогает браузерам правильно автоматически заполнять инпуты с помощью сохраненных данных.

Для полей электронного адреса используйте `autocomplete="username"`, поскольку `username` распознается менеджерами паролей в современных браузерах - даже если вам следует использовать `type="email"`, и вы можете захотеть использовать `id="email"` и `name="email"`. Для полей ввода пароля используйте соответствующие значения `autocomplete` и `id`, чтобы помочь браузерам различать новые и текущие пароли.

### Используйте `autocomplete="current-password"` и `id="current-password"` для существующих паролей

Важно: используйте `autocomplete="current-password"` и `id="current-password"` для инпута пароля в форме авторизации. Это сообщает браузеру, что вы хотите использовать текущий пароль, сохраненный для сайта.

Для формы авторизации:

```
<input type="password" autocomplete="current-password" id="current-password" …>
```

### Разрешите браузеру предлагать надежные пароли

Современные браузеры используют эвристики для принятия решения о показе UI менеджера паролей и предложения надежного пароля.

Встроенные в браузеры генераторы паролей избавляют пользователей и разработчиков от необходимости самостоятельно определять, что такое "надежный пароль". Поскольку браузеры могут безопасно хранить пароли и автоматически заполнять их по мере необходимости, пользователям не нужно запоминать или записывать пароли. Поощрение пользователей к использованию встроенных генераторов паролей также означает, что они с большей вероятностью будут использовать уникальный, надежный пароль на вашем сайте и с меньшей вероятностью будут повторно использовать пароль, который может быть или уже скомпрометирован в другом месте.

### Помогите предотвратить случайное отсутствие данных в полях

Важно: добавляйте атрибут `required` к полям электронного адреса и пароля. Современные браузеры автоматически запрашивают и устанавливают фокус при отсутствии данных.

```html
<input type="email" id="email" name="email" autocomplete="username" required>
<input type="password" id="password" name="password" autocomplete="current-password" required>
```

### Разрешите вставку пароля

Некоторые сайты запрещают вставку текста в поля пароля.

Запрет на вставку паролей раздражает пользователей, способствует созданию легко запоминающихся паролей (которые, следовательно, легче взломать) и, по данным таких организаций, как Национальный центр кибербезопасности Великобритании, может фактически снизить уровень безопасности. Пользователи узнают о запрете на вставку только после попытки вставить пароль, поэтому запрет на вставку паролей не предотвращает уязвимости буфера обмена.

### Поддержка и страховка

`type="email/tel/url"` поддерживаются всеми основными браузерами с 29.07.2015. `inputmode` - с 07.12.2021.

Автозаполнение - это прогрессивное улучшение. В браузерах без поддержки автозаполнения пользователям просто нужно будет вводить данные вручную. Семантические ограничения HTML (такие как `type`, `inputmode` и `required`) по-прежнему будут корректно функционировать для валидации пользовательского ввода и предоставления правильной виртуальной клавиатуры.
