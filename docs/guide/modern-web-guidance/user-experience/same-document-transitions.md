# Переходы в одном документе

## Проблема

Веб-сайты часто предоставляют несколько отображений (views) объекта, например, список товаров, а затем подробную страницу для каждого товара. Навигация между двумя отображениями часто ощущается несвязанной. Когда пользователь кликает по иконке товара для просмотра его подробностей, иконка исчезает и новое большое изображение появляется мгновенно где-то на экране. Это отсутствие преемственности затрудняет пользователям отслеживание отношений между элементами.

## Решение

API переходов отображений (View Transitions) позволяет определять пары элементов, существующие в разных состояниях до и после перехода. При запуске перехода с помощью `document.startViewTransition()` в одностраничном приложении (SPA), браузер определяет эти элементы по их уникальным `view-transition-name`. Затем он автоматически вычисляет разницу в их положении, размерах и стилях и плавно анимирует их от старого состояния к новому. Этот переход возникает в верхнем слое, над элементами с самыми высокими значениями `z-index`.

## Реализация

### Шаг 1: оборачивание изменений состояния в `startViewTransition()`

Для одностраничных приложений или простых изменений состояния оборачиваем логику обновления DOM в `document.startViewTransition()`. Браузер захватывает снимок (snapshot) текущего состояния, запускает обновление и затем захватывает новое состояние:

```javascript
function navigate(view) {
  document.startViewTransition(() => updateDOM(view));
}
```

### Шаг 2: присвоение одинаковых названий перехода

Используем CSS-свойство `view-transition-name` для указания браузеру, какие элементы должны быть преобразованы. Название может быть любым (кроме `none`). Важно: должен быть ровно 1 элемент до и после с определенным `view-transition-name`. Если таких элементов 2 и более, DOM обновляется к новому состоянию мгновенно, без перехода.

Можно использовать несколько `view-transition-name` для преобразования нескольких пар элементов. Например, можно преобразовать изображение и заголовок товара с помощью отдельных переходов.

Поскольку в отображении списка имеется несколько элементов, им нельзя присвоить одинаковые `view-transition-name`. Это можно решить двумя способами в SPA.

1. Динамичная страница подробностей: присваиваем каждому элементу списка уникальное `view-transition-name`, а затем динамически применяем это название совпадающему элементу на странице подробностей при выборе элемента списка:

```css
#product-1 { view-transition-name: p1 }
#product-2 { view-transition-name: p2 }
#product-3 { view-transition-name: p3 }
```

```js
function updateDOM(clickedTransitionName){
  const hero = document.getElementById("hero");
  hero.style.viewTransitionName = clickedTransitionName;
}
```

2. Динамичный элемент списка: присваиваем элементу на странице подробностей `view-transition-name` и применяем это название к элементу на странице списка при его выборе. Удаляем `view-transition-name` у элемента на странице списка при возвращении на нее.

Элемент `#hero` на странице подробностей и выбранный элемент `.thumbnail`  на странице списка делят одно `view-transition-name`:

```css
#hero{
  view-transition-name: hero;
}
.thumbnail.selected {
  view-transition-name: hero;
}
```

При клике по иконке, нужно подготовить отображение списка путем присвоения `view-transition-name` с помощью селектора класса `.selected` и выполнить любые изменения DOM до начала перехода.

Затем можно вызвать `document.startViewTransition()` и применить изменения к переходу страницы от подробностей к отображению списка.

После возвращения к отображению списка, нужно очистить классы перехода отображения для предотвращения ошибки следующей навигации. Эту очистку можно выполнить после разрешения промиса `finished`:

```javascript
// Функция, вызываемая при клике по иконке товара
function goFromListToDetail(e){
  e.currentTarget.classList.add("selected");
  const hero = document.getElementById("hero");
  const bgColor = getComputedStyle(e.currentTarget).backgroundColor;
  hero.style.background = bgColor;

  // Запускаем переход, проверяя поддержку
  if (!document.startViewTransition) {
    document.body.classList.add("detail");
    // Переводим фокус на новый заголовок для озвучивания контекста и сохранения логики фокусировки с помощью клавиатуры
    document.getElementById("detail-heading")?.focus();
    return; // Останавливаем выполнение функции, если переходы отображения не поддерживаются
  }
  const transition = document.startViewTransition(() => {
    document.body.classList.add("detail");
  });
  // Переводим фокус после разрешения перехода отображения
  transition.finished.finally(() => {
    document.getElementById("detail-heading")?.focus();
  });
}

// Функция, вызываемая при возврате от страницы подробностей к отображению списка
function goFromDetailToList() {
  if (!document.startViewTransition) {
    document.body.classList.remove("detail");
    document.getElementById("list-heading")?.focus();
    return;
  }
  const transition = document.startViewTransition(() => {
    document.body.classList.remove("detail");
  });
  // Очищаем отображение списка и переводим фокус
  transition.finished.finally(() => {
    // Переводим фокус обратно на отображение списка
    document.getElementById("list-heading")?.focus();
    // Удаляем класс `selected` для удаления всех `view-transition-name`
    document.querySelectorAll(".selected").forEach(
      (element) => {
        element.classList.remove("selected");
      },
    );
  });
}
```

Выбор конкретного метода зависит от случая использования. Динамичный элемент списка требует меньше повторяющегося CSS, но больше ручной очистки с помощью JavaScript.

### Шаг 3: исправление "растягивания" соотношения сторон

По умолчанию, браузер плавно преобразует старые и новые снимки внутри группы, которая растягивается так, чтобы соответствовать обоим. При выполнении перехода текста, следует установить для ширины текстового элемента значение `fit-content` как в старом, так и в новом представлении, чтобы соотношение сторон переходного элемента было стабильным:

```css
#list-page .title {
  width: fit-content;
}

#detail-page #title {
  width: fit-content;
}
```

При выполнении перехода элементов, у которых меняется соотношение сторон, может потребоваться установить высоту старого и нового отображение в 100%:

```css
::view-transition-old(hero),
::view-transition-new(hero) {
  height: 100%;
}
```

Эти псевдоэлементы являются снимками живых элементов, поэтому можно также использовать объявления `object-fit` и `object-position` для большего контроля эффекта перехода.

## Лучшие практики

- Не определяйте слишком много переходов. Делайте распределенными только те элементы основного контента, которые активно отслеживает пользователь (например, основные изображения, заголовки).
- Удаляйте временные значения `view-transition-name` после окончания перехода во избежание побочных эффектов в будущих переходах.
- Не выполняйте переходы элементов с активными анимациями. Переходы отображения оперируют снимками, поэтому любые анимации будут приостановлены во время перехода.
- Учитывайте предпочтения пользователя по снижению движения с помощью медиа-запроса `prefers-reduced-motion`:

```css
@media (prefers-reduced-motion: reduce) {
  ::view-transition-group(*),
  ::view-transition-old(*),
  ::view-transition-new(*) {
    animation: none !important;
  }
}
```

## Поддержка и страховка

Переходы отображения поддерживаются всеми основными браузерами с 14.10.2025.

API переходов отображения спроектировано для прогрессивного улучшения. Браузеры без поддержки просто выполняют обновление DOM мгновенно, без анимации:

```javascript
function navigate(){
  if (!document.startViewTransition) {
    // Резерв: просто обновляем DOM
    updateDOM();
  } else {
    document.startViewTransition(() => updateDOM());
  }
}
```
