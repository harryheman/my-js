# Руководство по конфиденциальности

Разработчики веб-приложений должны расценивать приватность как фундаментальное архитектурное требование, а не просто как галочку о соблюдении законодательства. Поскольку веб-экосистема переходит от пассивного отслеживания к явному с согласия пользователя, создание приложений, сохраняющих конфиденциальность, имеет решающее значение для доверия и безопасности пользователей.

В этом документе представлены общие принципы и подробные практические рекомендации с примерами кода для веб-разработчиков.

## Общие принципы

Основные темы, которые должны определять подход к конфиденциальности в веб-разработке:

1. Асимметрия автоматизации и обеспечение конфиденциальности: не перекладывайте бремя защиты конфиденциальности на пользователя (работа по обеспечению конфиденциальности). Не перегружайте пользователей сложными модальными окнами с запросами разрешений (асимметрия автоматизации). Пользователи имеют ограниченное время и внимание; перекладывать на них работу по обеспечению конфиденциальности часто неэффективно и вызывает утомление. Системы должны обеспечивать конфиденциальность по умолчанию.
2. Минимизация данных: "если у вас нет данных, вы не можете их потерять". Собирайте только необходимый минимум данных для текущей задачи. Сокращение хранилища данных снижает риск взлома и повышает доверие пользователей.
3. Ограничение по целям: данные, собранные для одной цели, не должны использоваться для другой цели без нового согласия пользователя. Перепрофилирование данных без явного согласия нарушает доверительные отношения с пользователем.
4. Прозрачность по умолчанию: честно и ясно объясняйте, зачем собираются данные, куда они отправляются и как долго хранятся. Прозрачность укрепляет доверие и может стать уникальным аргументом в пользу вашего приложения.
5. Надежное агентство: рассматривайте свое приложение как агента, действующего в интересах пользователя. Это означает, что приложение должно защищать пользователей от навязчивого поведения, ненужного раскрытия данных и выступать в качестве лояльного доверенного лица по отношению к пользователю, а не служить интересам третьих лиц.

## Подробные рекомендации

### 1. Минимизация данных и ограничение по целям

Сокращение объема собираемых данных и строгое ограничение их использования - наиболее эффективный способ защитить конфиденциальность пользователей.

#### Хорошо

* Собирайте данные с минимально необходимой степенью детализации. Если вам нужно знать только то, находится ли пользователь в определенной возрастной группе (например, 18–34), запрашивайте только его возрастную группу, а не точную дату рождения.
* Предоставляйте возможность оформления заказа для гостей электронной коммерции, чтобы избежать принудительного создания учетной записи, что сокращает сбор данных и повышает вероятность отказа от покупок.
* Удаляйте данные сразу после достижения цели их сбора.
* Используйте такие методы, как "фаззинг" (fuzzing) или добавление шума к данным (дифференциальная конфиденциальность) при сборе совокупной статистики.

#### Плохо

* Не собирайте данные "на всякий случай, вдруг пригодятся в будущем".
* Не используйте данные, собранные для одной цели (например, проверки безопасности), для другой цели (например, маркетинга) без явного согласия пользователя.

#### Пример кода

Вместо того, чтобы спрашивать точную дату рождения:

```html
<label for="age-bracket">Возрастная группа:</label>
<select id="age-bracket" name="age-bracket">
  <option value="18-34">18-34</option>
  <option value="35-49">35-49</option>
  <option value="50+">50+</option>
</select>
```

### 2. Прозрачность и доверие

Повышайте доверие, открыто рассказывая о своих методах работы с данными и предоставляя пользователям простые способы контроля своих данных.

#### Хорошо

* Предоставляйте инлайн-объяснения того, почему запрашиваются данные. Размещайте объяснение непосредственно рядом с полем ввода.
* Указывайте четкую причину и контекст запроса мощных возможностей браузера (например, камера, местоположение).
* Рассмотрите возможность использования элемента `<permission>`, если он поддерживается, чтобы сделать запросы разрешений декларативными, инициируемыми пользователями и действовать как посредники данных.
* Используйте заголовок `Clear-Site-Data` при выходе пользователя из системы, чтобы гарантировать, что в браузере не останется устаревших данных.
* Удаление аккаунта должно быть таким же простым, как его создание.

#### Плохо

* Не скрывайте обоснования сбора данных в длинных сложных политиках конфиденциальности.
* Не используйте "грязные" приемы, чтобы обманом заставить пользователей дать согласие.

#### Примеры кода

Инлайн-прозрачность (HTML):

```html
<div>
  <label for="phone">Номер телефона (опционально)</label>
  <input id="phone" type="tel" name="phone">
  <a href="#phone-help">Зачем нам это нужно?</a>
  <aside id="phone-help">
    Мы используем ваш номер телефона только для отправки кодов двуфакторной аутентификации для обеспечения безопасности аккаунта.
  </aside>
</div>
```

`Clear-Site-Data` при выходе из системы (ответ HTTP):

```http
HTTP/1.1 200 OK
Clear-Site-Data: "*"
```

Примечание: при очистке кеша не отправляйте этот ответ на главную страницу, чтобы предотвратить блокировку рендеринга UI на медленных устройствах; запускайте очистку кэша в фоновом режиме.

Встроенный контроль разрешений (HTML):

```html
<!-- Декларативный элемент permission с резервом -->
<permission type="geolocation" onpromptdismiss="updateMap()">
  <!-- Резерв для браузеров без поддержки -->
  <button onclick="navigator.geolocation.getCurrentPosition(updateMap)">
    Использовать мое местонахождения
  </button>
</permission>
```

### 3. Безопасность и обработка данных для конфиденциальности

Конфиденциальность опирается на безопасность. Уязвимости в приложении или небезопасное хранилище приводят к нарушению конфиденциальности.

#### Хорошо

* Удаляйте персональные данные из логов приложений. Используйте автоматическое маскирование для email, токенов и идентификаторов.
* Используйте флаг `HttpOnly` для куки, хранящих идентификаторы сессий для блокировки доступа к ним с помощью клиентских скриптов.
* Реализуйте ограничение количества запросов (rate limit) на конфиденциальных конечных точках (например, при поиске пользователей или просмотрах профилей), чтобы предотвратить массовую утечку данных.
* Используйте CHIPS (Cookies Having Independent Partitioned State - куки с независимым разделенным состоянием) путем добавления атрибута `Partitioned` для внедрений 1:1, которые не разделяют состояние на сайтах верхнего уровня.

#### Плохо

* Не храните чувствительные токены или персональные данные в `localStorage`, поскольку оно доступно встроенным скриптам.
* Не полагайтесь на неразделенные (unpartitioned) `SameSite=None` куки.

#### Примеры кода

Безопасный куки сессии (HTTP):

```http
Set-Cookie: session_id=xyz123; Secure; HttpOnly; SameSite=Lax
```

CHIPS куки (HTTP):

```http
Set-Cookie: theme_pref=dark; SameSite=None; Secure; Path=/; Partitioned; HttpOnly
```

### 4. Аудит сторонних скриптов и ресурсов

Сторонние скрипты и ресурсы являются частым источником утечек данных. Помните, что ответственность за третьих лиц, которых вы привлекаете в свое приложение, несете вы.

#### Хорошо

* Проводите регулярные проверки сетевых запросов на предмет того, какие данные собирают третьи стороны.
* Используйте шаблон фасада для тяжелых встраиваний (например, YouTube или TikTok). Отображайте статическую миниатюру и загружайте интерактивный iframe только после клика пользователя.
* Используйте для встраивания варианты, обеспечивающие конфиденциальность, если они доступны (например, `youtube-nocookie.com`).
* Замените тяжелые SDK для обмена информацией в социальных сетях простыми статическими HTML-ссылками, которые не отслеживают пользователей.

#### Плохо

* Не предполагайте, что третья сторона обеспечивает конфиденциальность только потому, что она популярна.
* Не загружайте сторонние скрипты на страницы, где обрабатываются конфиденциальные данные без крайней необходимости.

#### Примеры кода

Социальный обмен с сохранением конфиденциальности (HTML):

```html
<!-- Не требуется JS SDK -->
<a href="https://x.com/intent/tweet?text=Check%20this%20out&url=https%3A%2F%2Fexample.com"
   rel="noopener" target="_blank">
   Поделиться в X
</a>
```

Шаблон фасада для видео (HTML/JS):

```html
<div id="video-container" data-video-id="abc123">
  <img src="https://img.youtube.com/vi/abc123/maxresdefault.jpg" alt="Play Video" id="play-btn">
</div>

<script>
document.getElementById('play-btn').addEventListener('click', function() {
  const container = document.getElementById('video-container');
  const videoId = container.dataset.videoId;
  container.innerHTML = `<iframe src="https://www.youtube-nocookie.com/embed/${videoId}?autoplay=1" allowfullscreen></iframe>`;
});
</script>
```

### 5. Заголовки HTTP, обеспечивающие конфиденциальность

Используйте стандартные заголовки HTTP, чтобы указать браузеру обеспечить соблюдение границ конфиденциальности.

#### Хорошо

* Используйте `Permissions-Policy` для отключения мощных возможностей браузера (таких как камера, микрофон, геолокация) по умолчанию, включая их только по необходимости.
* Устанавливайте строгий `Referrer-Policy` для предотвращения утечки чувствительных параметров URL к третьим лицам.

#### Примеры кода

Строгий `Referrer-Policy` (HTTP):

```http
Referrer-Policy: strict-origin-when-cross-origin
```

Строгий `Permissions-Policy` (HTTP).

Отключаем мощные возможности браузера для всех источников по умолчанию:

```http
Permissions-Policy: geolocation=(), camera=(), microphone=(), accelerometer=()
```

### 6. Снятие отпечатков пальцев (fingerprinting) и сокращение числа пользовательских агентов

Избегайте методов, которые пытаются скрытно идентифицировать пользователей на основе конфигурации их устройств. Фингерпринтинг лишает пользователя контроля, поскольку он основан на неизменных характеристиках и происходит незаметно, не позволяя пользователям отказаться от участия или очистить свои данные.

#### Хорошо

* Используйте обнаружение функций (например, директиву `@supports` в CSS) вместо анализа агента пользователя, чтобы определить, поддерживает ли браузер конкретную возможность.
* Используйте "Клиентские подсказки пользовательского агента" (UA-CH), если это поддерживается браузером, когда целью является конкретное устройство.

#### Плохо

* Не используйте отрисовку на холсте (canvas), списки шрифтов или перечисления аудио/видеоустройств для создания отпечатка устройства.
* Не предполагайте полную корректность традиционной строки `navigator.userAgent`.

#### Примеры кода

Определение поддержки (JavaScript):

```javascript
// Хорошо: проверка существования API
if ('IntersectionObserver' in window) {
  const observer = new IntersectionObserver(...);
} else {
  // Резерв
}

// Плохо: анализ UA
// if (navigator.userAgent.includes("Chrome/100")) ...
```

Клиентские подсказки пользовательского агента (JavaScript):

```javascript
if (navigator.userAgentData) {
  navigator.userAgentData.getHighEntropyValues(["platformVersion", "architecture"])
    .then(ua => {
      console.log(ua.platformVersion);
    });
}
```

### 7. Права на данные и контроль пользователей

Предоставьте пользователям возможность реализовать свои права в отношении своих личных данных.

#### Хорошо

* Предоставляйте пользователям четкие механизмы доступа ко всем данным, которые вы о них собрали.
* Реализуйте автоматические или простые ручные процессы для удаления данных.
* Предоставьте пользователям возможность исправлять неточную информацию, связанную с их личностью.

#### Плохо

* Не усложняйте процесс удаления аккаунта и не требуйте от пользователей обращения в службу поддержки, если регистрация была простой.
* Не предпринимайте ответных мер против пользователей, которые реализуют свои права на данные, отказывая в доступе к определенным службам.
